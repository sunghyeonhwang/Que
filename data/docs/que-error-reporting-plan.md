# Que 에러 표시·리포팅 기획

마지막 업데이트: 2026-07-02
배경: Vercel 배포 후 팀원 8명에게 베타 테스트를 요청할 예정. 테스터가 에러를 만나면 ① 이해할 수 있는 화면을 보고 ② 개발자가 **반드시** 그 에러를 인지해야 한다. "유저가 말 안 하면 모르는 에러"가 0건이어야 베타가 의미 있다.

## 1. 에러의 두 종류 — 다르게 다룬다

| 종류 | 예 | 표시 | 리포팅 |
| --- | --- | --- | --- |
| **예상된 거부** (도메인 규칙) | 권한 없음, 사유 누락, 담당자 없는 Action 확정 | 이미 구현됨 — 토스트로 한국어 사유 표시 (`QueRuleError` → `{ok:false}`) | **불필요** — 정상 동작이다. 리포팅하면 노이즈만 쌓인다 |
| **예상 못 한 실패** (버그) | 서버 500, 렌더링 crash, fetch 실패, 처리 안 된 예외 | 현재 **Next.js 기본 에러 화면(영어, 정보 없음)** — 작업 필요 | 현재 **없음** — 작업 필요 |

이 문서는 두 번째 종류를 다룬다.

## 2. 사용자에게 보여줄 것 (표시)

### 2.1 에러 화면 — `error.tsx` / `global-error.tsx`

Next.js의 route segment error boundary를 사용한다. 만들 것:

- `apps/web/src/app/(app)/error.tsx` — 앱 셸은 유지한 채 본문만 에러 화면으로 교체
- `apps/web/src/app/global-error.tsx` — 셸 자체가 죽었을 때의 최후 방어선

화면 구성 (운영 도구 톤, 과장 없이):

```
문제가 생겼습니다.
이 오류는 자동으로 개발팀에 전달됐습니다.

오류 코드: {digest}          ← Next.js가 주는 에러 digest (스택트레이스는 절대 노출 안 함)

[다시 시도]  [오늘 화면으로]

(선택) 무엇을 하다가 문제가 생겼는지 한 줄 남겨주시면 해결에 큰 도움이 됩니다:
[__________________________] [보내기]
```

핵심 원칙:
- **"자동으로 전달됐습니다"가 거짓말이면 안 된다** — 3장의 리포팅이 먼저 동작해야 이 문구를 쓴다.
- 오류 코드(digest)를 보여주는 이유: 테스터가 "아까 그 에러"라고 말할 때 개발자가 리포트와 대조할 수 있는 유일한 열쇠다.
- 기술 정보(스택, 파일 경로)는 절대 노출하지 않는다.

### 2.2 조작 실패 토스트의 일관화

서버 액션의 `{ok:false}` 토스트는 유지하되, **예상 못 한 예외**(rethrow 경로)가 클라이언트에서 터지면 지금은 조용히 죽는다. 각 클라이언트 컴포넌트의 `startTransition` 내부를 try/catch로 감싸 "알 수 없는 오류가 발생했습니다 (자동 신고됨)" 토스트 + 리포팅을 태운다. 공통 래퍼 훅(`useSafeAction`)으로 중복을 없앤다.

## 3. 리포팅 채널 (결정 필요)

### 권장: Sentry

| 후보 | 장점 | 단점 | 판단 |
| --- | --- | --- | --- |
| **Sentry** (권장) | Vercel 공식 통합(설정 몇 분), 서버+클라이언트+source map, digest·사용자·URL 자동 수집, 이슈 그루핑/알림(이메일·Slack), 무료 티어로 8인 베타 충분 | 외부 서비스 의존, PII 스크러빙 설정 필요 | ✅ 채택 권장 |
| Vercel 로그만 | 추가 설정 없음 | **알림이 없다** — 들어가서 봐야만 안다. "반드시 리포팅" 요건 미충족. 클라이언트 에러는 아예 안 잡힘 | ❌ 단독으론 부족 |
| 자체 error_logs 테이블 + Slack 웹훅 | 완전 통제, Que 철학(ChangeLog)과 일관 | 직접 구현·유지보수, 그루핑/소스맵 없음, Slack 앱은 현재 생성 불가 상태 | 2차 보강 옵션 |

### 수집 지점 4곳 (하나라도 빠지면 "반드시"가 깨진다)

1. **서버(요청 처리 중)**: `instrumentation.ts`의 `onRequestError` 훅 — 서버 컴포넌트/서버 액션/route handler의 처리 안 된 예외 전부. `withApi`의 rethrow 경로도 여기서 잡힌다.
2. **클라이언트(렌더링)**: error boundary(`error.tsx`)에서 `Sentry.captureException` + 전역 `onerror`/`unhandledrejection`(SDK 자동).
3. **클라이언트(조작)**: 2.2의 `useSafeAction` 래퍼에서 캡처.
4. **MCP/CLI**: 별도 SDK 없이 — 서버 에러는 1번이 잡고, 클라이언트측(네트워크 등) 실패는 stderr 표준 출력으로 충분 (베타 테스터는 웹만 쓴다).

### 리포트에 담을 컨텍스트

- 에러 digest, URL/라우트, 현재 mock 사용자 id(누가 겪었나), `via`(web/mcp/cli), 앱 버전(커밋 SHA — Vercel env `VERCEL_GIT_COMMIT_SHA`)
- **PII 스크러빙 필수**: `beforeSend`에서 계좌번호 패턴(`\d{2,6}-\d{2,6}-\d{2,10}`)과 금액 필드를 마스킹. 결제 페이로드가 breadcrumb에 남지 않게 fetch body 수집은 끈다.

### 알림 (인지 보장)

- 1단계: Sentry 기본 — 새 이슈 발생 시 **이메일 알림** (즉시 가능)
- 2단계: 사무실 복귀 후 Sentry↔Slack 통합으로 채널 알림 (Slack 앱 이슈와 함께 해결)

## 4. 사용자 피드백 (에러가 아닌 "이상해요"용)

에러 화면의 한 줄 입력(2.1)과 별개로, 베타 기간에는 **모든 화면에서 접근 가능한 "피드백" 버튼**(사이드바 하단)을 둔다 → 텍스트 + 현재 화면 URL + 사용자 id를 자체 API(`/api/feedback` → 당장은 Sentry user feedback으로 전송)로 수집. 렌더링이 멀쩡해도 "숫자가 이상해요" 같은 제보를 받는 통로다.

## 5. 작업 목록과 순서 (배포 전 필수 = P0)

| # | 작업 | 우선순위 | env 필요? |
| --- | --- | --- | --- |
| 1 | `error.tsx` + `global-error.tsx` (한국어, digest, 다시 시도) | P0 | 아니오 |
| 2 | `useSafeAction` 래퍼 — 조작 실패 토스트 일관화 + 캡처 지점 | P0 | 아니오 |
| 3 | Sentry 연동 (`@sentry/nextjs`, instrumentation, beforeSend 스크러빙) | P0 | **DSN 필요** (Sentry 프로젝트 생성 — 무료, 5분) |
| 4 | 에러 화면 한 줄 피드백 전송 | P1 | 3 이후 |
| 5 | 사이드바 피드백 버튼 + `/api/feedback` | P1 | 3 이후 |
| 6 | Sentry↔Slack 알림 | P2 | 사무실 복귀 후 |

1·2번은 env 없이 지금 구현 가능하다 (Sentry 미연동 상태에서는 콘솔 로깅으로 동작하고, DSN이 오면 켜지는 구조로 만든다).

## 6. 결정 요청

1. **Sentry 채택** 여부 (무료 티어, 계정 하나 필요 — DSN만 주면 연동은 코드 한 줄)
2. 에러 화면의 한 줄 피드백 — 1단계부터 넣을지 (P1로 미룰지)
3. 베타 공지문에 "오류 코드 알려주기" 안내 포함 여부
