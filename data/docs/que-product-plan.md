# Que 제품 기획서

## 제품 정의

Que는 **캘린더 UI를 가진 팀 작업 상태 관리 도구**다.

기존 회사 캘린더, 프로젝트 마일스톤, 개인 작업, 작업 상태를 한 화면에 모아 8명 규모 팀이 오늘의 업무 흐름과 병목을 빠르게 확인하도록 돕는다. Que의 목적은 직원 감시가 아니라, 반복적인 확인 질문을 줄이고 막힌 일을 더 빨리 드러내는 것이다.

## 한 문장 설명

> Que는 팀원에게 계속 물어보지 않아도 오늘의 업무 흐름, 변경된 일정, 막힌 일을 캘린더에서 바로 알 수 있게 해주는 팀 작업 상태 캘린더다.

## 해결하려는 문제

- 회사 캘린더에는 회의와 마감은 있지만 실제 작업 진행 상태가 없다.
- 팀원이 지금 무엇을 하는지 확인하려면 직접 물어봐야 한다.
- 작업이 밀렸는지, 문제가 생겼는지, 완료됐는지 기록이 남지 않는다.
- 프로젝트 마일스톤과 개인 작업이 분리되어 있어 마감 위험을 빨리 알기 어렵다.
- 캘린더에 작업을 등록하는 과정이 번거롭다.
- 문제 발생, 홀드, 일정 변경 같은 상태가 개인 메신저에 흩어진다.

## 대상 사용자

### 팀원

- 오늘 내 일정과 작업을 확인한다.
- 작업 시작 시점에 간단히 상태를 응답한다.
- 문제 발생이나 일정 변경을 빠르게 공유한다.

### 대표/관리자

- 팀 전체의 오늘 업무 흐름을 확인한다.
- 문제 발생, 홀드, 마감 임박 작업을 먼저 본다.
- 사람별 업무량과 일정 충돌을 확인한다.

### 프로젝트 담당자

- 프로젝트별 마일스톤과 연결된 작업을 본다.
- 병목 작업과 담당자를 확인한다.
- 일정 변경 이력을 추적한다.

## 핵심 원칙

1. **묻지 않아도 보인다**
   - 누가 오늘 무엇을 하는지, 어디가 막혔는지 한 화면에서 보여준다.

2. **상태 변경은 3초 안에 끝난다**
   - 작업중, 완료, 문제발생, 시간변경 같은 주요 상태는 원터치로 처리한다.

3. **감시가 아니라 병목 공유다**
   - 근무 추적이 아니라 업무 진행, 충돌, 도움 요청을 중심으로 보여준다.

4. **캘린더는 입력 방식이 아니라 업무 맥락이다**
   - 내부 데이터는 일정, 작업, 마일스톤, 상태 로그로 구분하고 화면에서 캘린더처럼 표현한다.

5. **개인이 관리하고 팀이 변경을 본다**
   - 각 사용자는 로그인 후 자신의 작업과 업무 일정을 직접 생성, 수정, 삭제한다.
   - 업무에 영향을 주는 변경은 관리자와 관련 팀원이 볼 수 있도록 변경 내역에 기록한다.
   - 개인 비공개 일정은 상세 내용을 보호하되, 업무 충돌 판단에 필요한 시간 점유 여부는 표시할 수 있다.

## 주요 기능

### 1. 회사 캘린더 연동

- Google Calendar 캘린더를 가져온다. (2026-07-03 확정 — Outlook 지원은 범위 밖)
- 초기 버전에서는 읽기 전용으로 표시한다.
- 회의, 외부 일정, 휴가, 회사 이벤트를 Que 캘린더에 함께 표시한다.
- 개인 비공개 일정은 제목을 숨기거나 `비공개 일정`으로 표시한다.

### 2. 마일스톤 표시

- 프로젝트별 주요 마감일을 캘린더 상단에 표시한다.
- 마일스톤은 일반 작업과 구분되도록 다이아몬드 아이콘 또는 굵은 상단 라인으로 표시한다.
- 연결된 작업이 지연되면 마일스톤 위험 상태를 보여준다.

### 3. 드래그 일정 이동

- 각 캘린더/프로젝트의 마일스톤과 Que 작업 일정은 캘린더에서 드래그로 이동할 수 있다.
- 주간 보기에서는 다른 날짜/시간 칸으로 이동할 수 있다.
- 월간 보기에서는 다른 날짜로 이동할 수 있다.
- 캘린더 화면에서는 해당 캘린더 안의 마일스톤과 작업을 날짜 칸 사이에서 이동할 수 있다.
- 이동 즉시 화면에 반영하고, 서버 저장이 실패하면 원래 위치로 되돌린다.
- 이동 내역은 변경 로그에 남기고, 관련 팀원과 관리자에게 보인다.
- 회사 캘린더에서 읽어온 외부 일정과 비공개 자리비움 일정은 기본적으로 드래그 이동하지 않는다.
- 마일스톤 이동이 연결 작업의 일정에 영향을 줄 경우, 연결 작업도 함께 이동할지 별도로 확인한다.

### 4. 작업 생성과 공유

- 작업명, 담당자, 프로젝트, 날짜/시간, 예상 소요 시간, 설명을 등록한다.
- 작업은 개인 캘린더와 팀 캘린더에 함께 표시된다.
- 자연어 입력을 지원하되 등록 전 확인 단계를 둔다.
- 로그인한 사용자는 자신의 작업과 업무 일정을 직접 관리한다.
- 본인이 변경한 일정과 작업은 팀 캘린더, 팀 현황판, 프로젝트 화면에 반영된다.
- 시간 변경, 담당자 변경, 상태 변경, 삭제는 변경 로그에 남긴다.

예시:

```text
내일 오후 3시에 황성현씨 상세페이지 QA 넣어줘
```

해석 결과:

```text
작업명: 상세페이지 QA
담당자: 황성현
일시: 2026년 7월 2일 15:00
프로젝트: 미지정
상태: 예정
```

### 5. 작업 상태 관리

MVP 상태값:

- 예정
- 진행중
- 완료
- 시간변경필요
- 홀드
- 문제발생
- 취소/필요없음
- 다른 프로젝트와 병합

`문제발생`과 `홀드`는 추가 정보를 반드시 받는다.

- 사유
- 다음 액션
- 도움 필요한 사람
- 다시 확인할 시간

### 6. 권한과 공개 범위

Que의 권한은 `누가 수정할 수 있는가`와 `누가 볼 수 있는가`를 분리해서 설계한다.

편집 권한:

- 본인: 자신의 작업과 업무 일정을 생성, 수정, 삭제할 수 있다.
- 프로젝트 담당자: 해당 프로젝트에 연결된 작업과 마일스톤을 조정할 수 있다.
- 관리자: 전체 작업과 마일스톤을 조정할 수 있으며, 변경 이력을 확인할 수 있다.
- 다른 팀원: 기본적으로 타인의 작업을 직접 수정하지 않고 댓글, 도움 요청, 상태 확인 요청만 할 수 있다.
- 드래그 이동도 수정 권한과 동일하게 처리한다.

공개 범위:

- 회사 일정: 팀 전체에 표시한다.
- Que 작업: 기본적으로 팀 전체에 표시한다. (2026-07-03 확정 — 기본값은 전체 공개, 프로젝트 단위 제한은 예외적으로만 사용)
- 프로젝트 작업: 팀 전체 또는 프로젝트 참여자에게 표시한다.
- 개인 업무 일정: 시간, 제목, 상태를 팀에 표시한다.
- 개인 비공개 일정: 팀원에게는 제목을 숨기고 `비공개 일정` 또는 `자리비움`으로 표시한다. **관리자는 예외적으로 원본 제목/사유를 상세 열람할 수 있다** (2026-07-03 확정).

변경 가시성:

- 업무 일정이나 작업의 변경은 관리자와 관련 팀원이 볼 수 있다.
- 변경 내역에는 누가, 언제, 무엇을 바꿨는지 기록한다.
- 변경이 마일스톤, 담당자, 일정 충돌, 문제발생 상태에 영향을 주면 Attention Queue에 올린다.

### 7. 자동 체크인

작업 시작 시점 또는 설정된 확인 시점에 담당자에게 상태를 묻는다.

예시:

```text
황성현님, 13:00 예정된 "상세페이지 QA" 상태를 알려주세요.
```

선택지:

- 작업중
- 완료
- 다른 작업 때문에 시간 변경 필요
- 문제발생
- 필요없어짐
- @다른프로젝트와 병합
- 나중에 답변

체크인 정책:

- 회의, 휴가, 외부 일정에는 묻지 않는다.
- Que에서 생성된 `작업`에만 묻는다.
- 이미 상태가 업데이트된 작업은 다시 묻지 않는다.
- 사용자는 방해금지 시간과 나중에 답변을 설정할 수 있다.

### 8. 팀 현황판

사람별, 시간대별 작업 상태를 캘린더형 보드로 보여준다.

보여줄 정보:

- 현재 진행중인 작업
- 오늘 예정 작업
- 문제발생 작업
- 홀드 작업
- 마감 임박 작업
- 응답 대기 작업
- 일정 충돌
- 최근 변경된 일정/작업

관리자 화면의 초점은 “누가 일하고 있나”가 아니라 “어디가 막혔나”여야 한다.

### 9. 작업량 히트맵

멤버별 작업량 편차를 색상과 그래프로 보여준다.

보여줄 정보:

- 멤버별 일자별 작업량
- 예정 작업 수
- 예상 소요 시간
- 문제발생/홀드 작업 가중치
- 마감 임박 작업 가중치
- 과부하 멤버
- 여유 있는 멤버

운영 규칙:

- 히트맵은 개인 평가가 아니라 업무 배분과 병목 조정을 위한 화면으로 설계한다.
- 단순 작업 개수만 세지 않고 예상 소요 시간, 마감 임박, 문제발생 상태를 함께 반영한다.
- 과부하 기준은 팀 설정에서 조정할 수 있어야 한다.
- 특정 멤버 셀을 클릭하면 해당 날짜의 작업 목록으로 이동할 수 있다.

### 10. 프로젝트 뷰

프로젝트 단위로 마일스톤, 작업, 담당자, 문제발생 항목을 확인한다.

구성:

- 프로젝트 개요
- 마일스톤 목록
- 연결된 작업
- 상태별 작업 목록
- 문제발생/홀드 항목
- 일정 변경 기록
- 병합된 작업

### 11. 결제/입금 확인

멤버가 결제할 내용을 등록하고, 입금 처리 후 진행 상태를 확인한다.

입력 필드:

- 제목
- 은행명
- 계좌번호
- 금액
- 내용
- 마감일
- 분류

상태:

- 대기
- 완료
- 취소

운영 규칙:

- 멤버는 본인이 필요한 결제/입금 요청을 등록한다.
- 관리자 또는 결제 담당자는 입금 처리 여부를 확인한다.
- 입금 처리 후 상태를 `완료`, `대기`, `취소`로 갱신한다.
- 상태 변경은 변경 로그에 남긴다.
- 마감일이 지난 대기 항목은 결제 화면 상단에 노출한다.
- 계좌번호와 금액은 민감 정보이므로 권한 있는 사용자에게만 상세 노출하는 설정을 둘 수 있다.

### 12. 회의록 업로드

Plaud Note에서 내보낸 Markdown 회의록을 Que에 업로드한다.

입력/관리 정보:

- 회의록 MD 파일
- 회의명
- 회의 일시 (날짜+시간)
- 프로젝트
- 참석자
- 업로드한 사람
- 공개 범위
- Action 추출 상태

운영 규칙:

- 업로드한 회의록은 원문을 보존한다.
- 회의록은 프로젝트와 연결할 수 있어야 한다.
- Action 후보를 만들 때 원문 문장과 파일명을 함께 연결한다.
- 민감한 회의록은 프로젝트 참여자 또는 관리자만 보도록 공개 범위를 설정할 수 있다.
- 기본값은 전체 팀 열람 가능이며, **회의록 단위로 좁은 열람 권한을 지정할 수 있어야 한다** (2026-07-03 확정 — 예: 연봉협상 회의록은 당사자와 대표만 열람).

### 13. Action Task 생성

회의록 MD에서 결정사항, 할 일, 이슈를 추출해 Task 후보로 만든다.

필수 확인 정보:

- Task 제목
- 담당자
- 마감일
- 프로젝트
- 원문 출처
- 상태

운영 규칙:

- 담당자와 마감일이 있는 항목만 바로 Task로 생성한다.
- 담당자 또는 마감일이 없으면 `확인 필요` 상태로 남긴다.
- 생성된 Task는 담당자의 캘린더와 Now 화면에 표시한다.
- Task에는 회의록 파일명과 원문 문장을 연결해 왜 생긴 일인지 추적한다.
- 잘못 추출된 후보는 무시하거나 보류할 수 있다.
- 추출 방식은 **규칙 기반과 LLM 기반을 모두 지원한다** (2026-07-03 확정) — 규칙 기반을 기본/폴백으로 먼저 구현하고, LLM 기반 추출을 옵션으로 추가한다. 어느 쪽이든 결과는 항상 `확인 필요` 확인 카드를 거친 뒤에만 Task로 확정된다.

### 14. Now 통합표

캘린더 일정과 Action Task를 표로 합쳐 오늘 바로 확인할 수 있게 한다.

포함 요소:

- 오늘 캘린더 일정
- 회의록에서 생성된 Action Task
- 담당자
- 마감일
- 현재 상태
- 출처
- 문제/홀드/담당자 누락 항목

Now는 개인 업무 화면인 `오늘`보다 운영자와 팀 공유 맥락에 더 가깝다. 회의에서 나온 일이 실제 캘린더와 작업 상태에 반영됐는지 확인하는 허브 역할을 한다.

### 15. 클라이언트(거래처)와 프로젝트 2단 분류

작업을 `거래처 → 프로젝트 → 작업`의 2단 분류로 정리한다. (2026-07-04 구현)

- 거래처(Client)는 이름, 상태(active/archived), 표시 순서를 가진다.
- 프로젝트는 하나의 거래처에 소속되거나, 내부 잡무처럼 어디에도 소속되지 않을 수 있다(clientId 없음).
- 작업은 프로젝트를 통해 거래처에 간접 연결된다. 작업에 거래처를 따로 지정하지 않는다.
- 작업 목록, 일정, 회의록, 성과 화면에서는 `거래처 · 프로젝트` 2단으로 표기한다. 공간이 좁은 곳은 프로젝트명만 표기한다.

클라이언트 전환 스위처:

- 모든 팀원이 사이드바 상단(모바일은 상단바)의 스위처에서 거래처를 선택할 수 있다.
- 거래처를 선택하면 작업 목록, 일정, 성과, 팀 현황이 해당 거래처의 것만 보인다. `전체 클라이언트`를 선택하면 필터가 해제된다.
- 성과와 홈에는 `○○ 기준` 배지를 표시해 지금 어떤 거래처 기준으로 보고 있는지 알린다.
- 회사 공통 일정은 어떤 거래처를 선택해도 항상 표시한다.
- 프로젝트가 없는 무소속 작업은 특정 거래처를 선택하면 숨긴다.
- 회의록, 결제, 알림 뱃지는 거래처 필터의 영향을 받지 않는다.

관리:

- 거래처와 프로젝트의 생성/편집은 관리자 전용 클라이언트 관리 화면에서 한다. (아래 "클라이언트 관리 화면" 참고)
- MCP/CLI에서는 클라이언트 목록 조회와 작업 조회의 `--client` 필터를 제공한다. (조회 전용)

### 16. 반복 업무와 마일스톤 관리

3차 버전으로 미뤘던 반복 업무 템플릿을 앞당겨 구현하고, 마일스톤 관리와 함께 `반복·마일스톤` 화면으로 제공한다. (2026-07-04 구현)

- 매주/매월 반복되는 정기 업무를 템플릿으로 등록하고 켜고 끌 수 있다.
- 켜진 템플릿은 다음 회차 Task를 자동 생성한다.
- 프로젝트 마일스톤을 생성/수정하고 기한과 위험 상태를 표시한다.
- 마일스톤 생성/수정은 프로젝트 담당자와 관리자만 할 수 있다. (기존 편집 권한 규칙과 동일)
- 태스크와 마일스톤은 직접 연결하지 않고 프로젝트 공유로 간접 연결한다.

### 17. 계정 보안과 액세스 토큰

- 비밀번호는 본인이 설정 화면에서 변경한다. 현재 비밀번호 확인을 거친다.
- 임시 비밀번호로 로그인하면 첫 로그인에서 비밀번호 변경을 강제한다.
- 관리자는 팀원의 비밀번호를 임시 비밀번호로 재설정할 수 있다.
- 로그인을 연속 실패하면 일정 시간 잠근다. (레이트리밋)
- MCP/CLI용 개인 액세스 토큰(PAT)은 설정 › 액세스 토큰에서 본인이 직접 발급/폐기한다.
- 토큰 평문은 발급 순간 1회만 표시하고, 서버에는 해시만 저장한다.

## 주요 화면 기획

### Now 화면

목적: 캘린더 일정과 회의록 기반 Action Task를 한 표에서 확인. 현행 IA에서는 `작업 목록` 메뉴의 Now 탭이다.

기본 진입점: 관리자/프로젝트 담당자. 화면 상단에 "회의 Action과 캘린더 일정이 연결됐는지 확인하는 팀 운영표" 한 줄 설명을 고정 표시한다.

포함 요소:

- 시간/마감일
- Calendar 또는 Action 구분
- 제목
- 담당자
- 상태
- 출처
- 문제/홀드/담당자 누락 요약
- 일정 충돌 지표

이 화면은 “회의에서 나온 일이 실제로 오늘의 업무와 연결됐는가”를 확인하는 운영표다. 캘린더를 보지 않아도 오늘 처리할 Action과 일정 충돌을 빠르게 확인할 수 있어야 한다.

### 오늘 화면

목적: 팀원이 하루에 한 번 반드시 확인하는 개인 업무 허브. 현행 IA에서는 `작업 목록` 메뉴의 오늘 탭이다.

기본 진입점: 팀원. 화면 상단에 "내 하루를 시작하는 개인 화면" 한 줄 설명을 고정 표시한다.

포함 요소:

- 오늘 내 일정
- 오늘 내 작업
- 지금 진행중인 작업
- 상태 응답이 필요한 항목
- 마감 임박 작업
- 내가 관련된 문제발생/홀드 작업

화면 구성 (2026-07-04 개편):

- 상단에 현황 KPI 한 줄을 고정하고, 타임라인·자동 체크인·하루 마감·주의 필요는 `현황 상세`로 묶어 접기/펼치기한다.
- 상세를 접은 상태에서는 작업 목록을 화면의 중심으로 크게 보여준다.
- 각 작업 행 오른쪽에 원형 완료 버튼을 둔다. 호버하면 채워지고, 클릭하면 완료 처리와 함께 폭죽 애니메이션을 보여준다. 다시 클릭하면 완료를 되돌린다.

### 캘린더 화면

목적: 회사 일정, 작업, 마일스톤을 한 캘린더에서 확인. 기존 `전체 캘린더`, `가로 캘린더`는 별도 메뉴가 아니라 이 화면의 뷰다. 현행 IA에서는 `일정` 메뉴다.

뷰 스위처 (상단 고정):

- 기본형: 시간축 격자, 내 일정 중심. 사람별/프로젝트별 필터 제공
- 전체 멤버: 사람별 행 × 요일별 열 (아래 "전체 멤버 뷰" 정의 참고)
- 타임라인: 날짜 가로축 (아래 "타임라인 뷰" 정의 참고)

기간 스위처:

- 일간
- 주간
- 월간

뷰 상태 규칙:

- 마지막 사용 뷰와 기간을 기억해 다음 방문 시 유지한다.
- 뷰와 기간을 URL에 반영해 특정 화면을 링크로 공유할 수 있게 한다.

표시 규칙:

- 회사 일정: 중립 색상
- 내 작업: 명확한 강조 색상
- 다른 사람 작업: 담당자 라벨
- 마일스톤: 별도 아이콘 또는 상단 띠
- 문제발생: 빨간 상태 표시
- 홀드: 노란 상태 표시
- 완료: 흐리게 처리
- 드래그 가능 항목: Que 작업, 캘린더/프로젝트 마일스톤
- 드래그 불가 항목: 외부 회사 캘린더 원본 일정, 비공개 자리비움

### 캘린더 — 전체 멤버 뷰

목적: 상세 패널이나 상태판 없이 전체 멤버의 일정표만 주간/월간 캘린더로 확인. `캘린더` 메뉴 안의 뷰로 제공한다.

포함 요소:

- 전체 멤버 범례
- 주간 보기: 사람별 행, 요일별 열
- 월간 보기: 날짜별 전체 멤버 일정 요약
- 회사 일정, Que 작업, 비공개 자리비움 표시
- 멤버별 색상 구분
- Que 작업과 마일스톤의 드래그 이동

이 화면은 관리자나 팀원이 “이번 주에 누가 언제 가능한지”를 빠르게 보는 용도다. 문제발생, 홀드, 변경 내역 같은 운영 정보는 팀 현황판에서 확인하고, 전체 캘린더에서는 일정표 자체에 집중한다.

### 캘린더 — 타임라인 뷰 (가로형)

목적: 일반적인 주간/월간 격자 대신 날짜를 가로축으로 두고 멤버와 캘린더 일정을 타임라인처럼 확인. `캘린더` 메뉴 안의 뷰로 제공한다.

포함 요소:

- 날짜 가로축
- 멤버별 일정 행
- 캘린더/프로젝트별 마일스톤 행
- 여러 날짜에 걸친 작업 막대
- 오늘 기준선
- 드래그 이동 가능한 Que 작업/마일스톤

이 화면은 “언제부터 언제까지 이어지는 작업인가”, “마일스톤이 어느 작업과 겹치는가”, “팀원별 업무가 특정 날짜에 몰려 있는가”를 보기 위한 화면이다. 일반 캘린더보다 프로젝트 운영과 일정 충돌 확인에 유리하다.

### 작업 상세 패널

목적: 페이지 이동 없이 작업을 빠르게 확인하고 수정.

포함 요소:

- 작업명
- 담당자
- 프로젝트
- 시간
- 상태
- 설명
- 체크리스트
- 관련 일정
- 상태 변경 기록
- 문제발생/홀드 상세 (사유, 다음 액션, 도움 필요한 사람, 다시 확인할 시간)
- 댓글/메모

주요 상태 버튼은 항상 노출한다.

```text
[진행중] [완료] [시간변경] [문제발생] [홀드]
```

### 팀 현황 화면

목적: 팀 전체의 오늘 업무 흐름과 병목 확인.

구성:

- 상단 요약: 진행중, 문제발생, 홀드, 마감 임박, 응답 대기
- 사람별 시간표
- 주의 필요 작업 목록
- 일정 충돌 목록
- 최근 변경 내역

### 히트맵 화면

목적: 멤버별 작업량이 어느 날짜에 몰려 있는지 한눈에 확인. 현행 IA에서는 `성과` 메뉴로 확장됐다 — KPI·완료율·기한초과·프로젝트 진행률을 함께 보여주고, 기간/월 선택은 URL로 구동한다. 개인 평가가 아니라는 원칙은 그대로다.

포함 요소:

- 멤버별/요일별 작업량 히트맵
- 작업량 강도 범례
- 멤버별 총 작업량 막대 그래프
- 과부하/여유 멤버 요약
- 문제발생/홀드 가중치 표시
- 캘린더/팀 현황으로 이동 가능한 작업량 셀

이 화면은 업무량이 많은 사람과 적은 사람을 빠르게 파악해서 일정 조정, 담당자 재배정, 마감 위험 대응에 쓰기 위한 현황판이다.

#### 성과 스코프 (직급별, 2026-07-06)

개인 평가가 아니라는 원칙을 유지하면서, **사람이 개별로 드러나는 위젯만** 직급(grade)으로 노출 범위를 좁힌다. grade는 DB user의 rank를 받는 `gradeForUser(user)`(대표=ceo, 관리=manager, 그 외=staff)로 판정한다. role(admin/member)은 쓰기 권한 전용이라 그대로 둔다.

- **직급 스코프를 적용하는 위젯**: 히트맵 rows(멤버별 부하), 부하표(팀 부하 현황).
  - 대표: 전원(8)
  - 관리자: 대표를 제외한 전원(7). 관리자끼리는 상호 노출.
  - 사원: 본인(1). 사원의 부하표는 프론트가 숨긴다.
- **grade와 무관하게 전원 동일**: KPI(총/진행/완료/기한초과 카드), 작업 완료율, 기한 초과 추이, 프로젝트 진행률. 단 **사원의 KPI·작업 성과 라인은 본인 스코프**(개인 대시보드 성격).
- **스코프는 세션 사용자의 grade에서만 유도한다. URL 파라미터로 넓힐 수 없다.** 메뉴/페이지(프론트)·페이지가 넘기는 세션 뷰어·데이터 계층의 viewer.id 재유도로 3중 방어한다(데이터 계층이 스코프를 다시 계산하므로 호출자가 남의 스코프를 요청해도 무의미).

### 직급별 홈 (2026-07-10 목표 명세 갱신)

홈은 grade에 따라 세 화면으로 조립하되, 동일 기능은 공통 `Home/*` 이름과 컴포넌트를 사용한다. 화면 표시명은 `오늘 요약`·`핵심 현황`·`우선 확인`·`작업 상태 확인`·`오늘 할 일`·`오늘 일정`·`프로젝트 현황`·`업무 흐름`·`업무 부하`·`처리 대기`로 고정한다(`확인 필요`는 회의록 Action의 고유 용어라 홈 카드명으로 쓰지 않는다). 역할 차이는 helper text·데이터 스코프·허용 액션으로만 표현한다. 상세 계약과 Figma 이름은 `data/docs/que-home-spec.md`를 따른다.

- **사원 홈 — `내 하루`**: 본인 스코프의 오늘 요약·핵심 현황·우선 확인·오늘 할 일·오늘 일정·업무 흐름. 응답 대기 자동 체크인은 있을 때만 **`작업 상태 확인` 리스트형 독립 카드**(행당 1작업·응답 버튼 7종)로 표시하고(2026-07-10 개정), 우선 확인에는 체크인을 제외한 본인 신호(기한초과·도움/확인 요청·확인 필요 Action)를 담는다. 현행 전역 `getAlerts`를 본인 신호라고 간주하지 않고 viewer-scoped selector를 둔다. **상단바 벨·알림 센터도 같은 selector를 쓴다(벨 개인화, 2026-07-10 결정)** — 벨·알림 센터·사원 우선 확인의 수치가 일치해야 한다.
- **관리자 홈 — `팀 운영`**: 오늘 요약(**관리자 variant는 `팀관리 AI 브리핑`** — AI 생성 승인 2026-07-10 대표와 통일, 관리자 스코프만 전송·온디맨드·고지·규칙 폴백)·팀 핵심 현황·우선 확인(문제/홀드/도움 요청/일정 충돌/상태 응답 대기, 대표 담당 문제·홀드 포함)·본인 오늘 할 일/일정·프로젝트 현황·업무 흐름·업무 부하·처리 대기. 업무 부하는 대표를 제외하며 타인 작업 직접 변경 대신 댓글·도움 요청·상태 확인 요청만 제공한다.
- **대표 홈 — `전사 현황`**: 오늘 요약(**대표 variant는 `경영 AI 브리핑`** — AI 생성 승인 2026-07-10, E-10 Gemini 재사용·온디맨드 버튼·전송 데이터 고지, 미생성 시 규칙 기반 폴백)·전사 핵심 현황·우선 확인(결정 필요)·본인 오늘 할 일/일정·프로젝트/클라이언트 현황·전사 업무 흐름·전 인원 부하·처리 대기.
  - **클라이언트별 현황**: 활성 거래처마다 소속 활성 프로젝트 수·평균 진행률·막힌 작업 수(문제/홀드)·열린 작업 수를 집계한다. 진행률 정의는 성과 화면과 같은 술어(done/전체, merged·cancelled 제외)를 써 화면 간 숫자를 맞춘다.
  - **자동 체크인 표현**: 사원에게는 조건부 `작업 상태 확인`, 관리자·대표에게는 `우선 확인` 안의 `상태 응답 대기`로만 표시한다. 자동 체크인을 별도 상시 홈 섹션으로 만들지 않는다.

### 회의록 화면

목적: Plaud Note MD 파일을 Que에 업로드하고 Action 추출 대상으로 관리. 현행 IA에서는 `확인필요` 메뉴의 탭이다.

포함 요소:

- MD 파일 업로드 영역
- 회의명
- 프로젝트
- 회의 일시 (날짜+시간)
- 참석자
- 업로드된 회의록 목록
- 추출 대기/Task 생성/확인 필요 상태
- Markdown 원문 미리보기

회의록 화면의 핵심은 원문 보존이다. AI나 규칙 기반 추출이 틀릴 수 있으므로, Action 후보와 원문 문장을 항상 함께 볼 수 있어야 한다.

### Action 화면

목적: 회의록에서 추출된 Task 후보를 실제 Que Task로 확정. 현행 IA에서는 `확인필요` 메뉴의 탭이다.

포함 요소:

- 원본 회의록 선택
- Task 후보 목록
- 담당자
- 마감일
- 원문 출처
- Task 생성/보류/무시 버튼
- 생성된 Task 목록

담당자와 마감일이 없는 후보는 자동 생성하지 않는다. 확인이 필요한 항목은 Now 화면과 팀 현황판에 표시해 회의 후 할 일이 사라지지 않게 한다.

### 프로젝트 화면

목적: 프로젝트 기준으로 마일스톤과 연결 작업을 추적.

구성:

- 프로젝트 상태
- 다음 마일스톤
- 캘린더 일정 그리드
- 진행중 작업
- 문제/홀드 작업
- 담당자별 할당
- 일정 변경 이력

캘린더 일정 그리드에서는 해당 캘린더의 마일스톤과 연결 작업을 날짜별로 배치하고, 드래그 이동 시 프로젝트 상태 이력과 팀 캘린더에 반영한다.

### 결제 화면

목적: 팀원이 결제할 내용을 등록하고, 입금 처리 후 진행 상태를 확인.

포함 요소:

- 결제 요청 등록 폼
- 제목, 은행명, 계좌번호, 금액, 내용, 마감일, 분류 입력
- 대기/완료/취소 상태 요약
- 결제 요청 목록
- 상태 변경 버튼
- 마감 임박/마감 초과 항목 표시

이 화면은 정산이나 송금 요청을 메신저에 흩어두지 않고, 누가 무엇을 언제까지 입금해야 하는지 명확히 남기기 위한 화면이다.

### 클라이언트 관리 화면

목적: 거래처와 프로젝트의 2단 분류를 관리. 관리자 전용. (2026-07-04 추가)

포함 요소:

- 거래처 목록 (이름, 상태 active/archived)
- 거래처 생성/편집
- 거래처 순서 변경 (드래그 또는 위/아래 버튼)
- 프로젝트 생성/편집 (이름, 상태, 담당자)
- 프로젝트의 클라이언트 재배정

운영 규칙:

- 관리자만 접근한다. 메뉴 노출, 화면 진입, 서버 처리의 3중으로 권한을 확인한다.
- 거래처 순서는 클라이언트 스위처와 목록의 표시 순서에 그대로 반영된다.

### 반복·마일스톤 화면

목적: 정기 반복 업무 템플릿과 프로젝트 마일스톤을 한 화면에서 관리. (2026-07-04 추가)

포함 요소:

- 반복 업무 템플릿 목록 (등록, 켜기/끄기)
- 켜진 템플릿의 회차 Task 자동 생성
- 프로젝트 마일스톤 목록 (생성/수정, 기한, 위험 상태 표시)

운영 규칙:

- 마일스톤 생성/수정은 프로젝트 담당자와 관리자만 할 수 있다.
- 태스크와 마일스톤은 직접 연결하지 않고 프로젝트 공유로 간접 연결한다.

### 설정 화면

알림 설정 등 설정 화면의 전체 기획은 개발 후순위지만, 아래 항목은 구현되어 제공 중이다. (2026-07-04)

모양:

- 테마: 라이트 / 다크
- 밀도: 기본 / 컴팩트
- 한글 폰트: SUIT(기본), Pretendard, Noto Sans KR, 페이퍼로지, 프리젠테이션, 시스템
- 영문 폰트: Inter Tight(기본), 페이퍼로지, 프리젠테이션, 시스템
- 모양 설정은 브라우저별(쿠키)로 저장한다. 같은 계정이라도 기기마다 다르게 둘 수 있다.

보안:

- 비밀번호 변경 (현재 비밀번호 확인 필수)

액세스 토큰 (MCP·CLI):

- 개인 액세스 토큰(PAT) 발급/폐기
- 평문은 발급 순간 1회만 표시하고, 서버에는 해시만 저장한다.

## UX 플로우

### 하루 시작 플로우

1. 사용자가 Que를 연다.
2. `오늘의 Que`에서 내 일정과 작업을 확인한다.
3. 미완료 작업이나 응답 대기 작업을 확인한다.
4. 필요하면 작업 시간을 수정하거나 상태를 바꾼다.

### 자연어 작업 생성 플로우

1. 사용자가 빠른 입력창에 자연어로 입력한다.
2. Que가 날짜, 시간, 담당자, 프로젝트, 작업명을 추출한다.
3. 확인 카드를 보여준다.
4. 사용자가 등록, 수정, 취소 중 하나를 선택한다.
5. 등록된 작업이 캘린더와 담당자 오늘 화면에 표시된다.

### 드래그 일정 이동 플로우

1. 사용자가 Que 작업 또는 캘린더 마일스톤을 드래그한다.
2. 다른 날짜/시간 칸에 놓는다.
3. 화면에서 즉시 새 위치로 이동한다.
4. Task 또는 Milestone의 시작/종료 시간 또는 마감일을 업데이트한다.
5. 변경 로그에 누가, 언제, 어디에서 어디로 이동했는지 기록한다.
6. 연결된 마일스톤, 일정 충돌, 체크인 시간이 있으면 재계산한다.
7. 관련 팀원과 관리자 화면에 변경이 반영된다.

### 자동 체크인 플로우

1. 작업 시작 시간이 된다.
2. 담당자에게 상태 질문이 전송된다.
3. 담당자가 객관식으로 응답한다.
4. 응답 결과가 작업 상태와 팀 현황판에 반영된다.
5. 문제발생/홀드인 경우 추가 정보를 받고 관련자에게 알린다.

### 문제발생 플로우

1. 담당자가 `문제발생`을 선택한다.
2. 사유, 도움 필요한 사람, 다음 확인 시간을 입력한다.
3. 해당 작업이 팀 현황판의 주의 필요 영역에 올라간다.
4. 프로젝트 담당자에게 알림이 간다.
5. 해결되면 상태 로그에 기록된다.

### 결제 요청 플로우

1. 멤버가 결제 화면에서 제목, 은행명, 계좌번호, 금액, 내용, 마감일, 분류를 입력한다.
2. 결제 요청은 기본 상태 `대기`로 생성된다.
3. 결제 담당자 또는 관리자가 입금 처리 여부를 확인한다.
4. 상태를 `완료`, `대기`, `취소` 중 하나로 변경한다.
5. 상태 변경 내역은 변경 로그에 기록된다.
6. 마감일이 지난 대기 항목은 결제 화면 상단에 노출한다.

### 회의록 업로드 플로우

1. 사용자가 Plaud Note에서 Markdown 파일을 내보낸다.
2. Que의 회의록 화면에 MD 파일을 업로드한다.
3. 회의명, 프로젝트, 회의일, 참석자를 확인한다.
4. 회의록은 Action 추출 대기 상태로 저장된다.
5. 업로드된 회의록은 원문 미리보기와 함께 목록에 표시된다.

### Action Task 생성 플로우

1. 사용자가 Action 화면에서 원본 회의록을 선택한다.
2. Que가 결정사항, 할 일, 이슈 문장을 Task 후보로 추출한다.
3. 후보별 담당자와 마감일을 확인한다.
4. 담당자와 마감일이 있는 후보는 Task로 생성한다.
5. 담당자 또는 마감일이 없는 후보는 확인 필요로 남긴다.
6. 생성된 Task는 캘린더, Now, 담당자 오늘 화면에 표시된다.

### Now 확인 플로우

1. 사용자가 Now 화면을 연다.
2. 캘린더 일정과 Action Task를 한 표로 확인한다.
3. 문제발생, 홀드, 담당자 누락, 오늘 마감 항목을 먼저 본다.
4. 필요하면 Action 화면으로 이동해 Task를 생성하거나 보류한다.
5. 캘린더 일정과 충돌하는 Action은 일정 조정 대상으로 표시한다.

### 클라이언트 전환 플로우

1. 사용자가 사이드바 상단(모바일은 상단바)의 클라이언트 스위처에서 거래처를 선택한다.
2. 작업 목록, 일정, 성과, 팀 현황이 해당 거래처의 프로젝트와 작업만 보여준다.
3. 성과와 홈에 `○○ 기준` 배지가 표시된다.
4. 회사 공통 일정은 그대로 표시되고, 프로젝트 무소속 작업은 숨겨진다.
5. 회의록, 결제, 알림 뱃지는 필터의 영향을 받지 않는다.
6. `전체 클라이언트`를 선택하면 필터가 해제된다.

## 알림 정책

알림은 적게 보내고, 보낼 때는 바로 처리 가능해야 한다.

### 알림 채널 (2026-07-02 확정)

**Slack을 기본 알림 채널로 사용한다.**

- 1단계 (MVP 직후): Slack Incoming Webhook으로 체크인 질문·문제발생 공유·마감 임박을 발송한다. 메시지에는 해당 항목으로 바로 여는 Que 딥링크를 담는다 (예: `/today`, 작업 상세).
- 2단계: Slack 앱(Bot)으로 전환해 메시지의 인터랙티브 버튼으로 체크인에 **Slack 안에서 바로 응답**한다. 응답은 core의 `answerCheckIn`을 거치므로 웹과 동일한 규칙·로그가 적용된다 (`via` 필드에 채널 기록).
- 앱 내 알림 화면은 Slack 발송 내역의 허브 역할을 겸한다 (프리뷰 수령 후 구현).
- 방해금지 시간대에는 Slack 발송을 보류하고 앱 내 알림에만 쌓는다.

보내는 알림:

- 작업 시작 체크인
- 문제발생/홀드 공유
- 도움 요청
- 마감 임박
- 일정 충돌
- 하루 요약
- 관련 작업의 시간/담당자/상태 변경
- 결제 요청 마감 임박/마감 초과
- 결제 요청 상태 변경
- 회의록 Action 후보 확인 필요
- 회의록에서 생성된 Task 담당자 지정

보내지 않는 알림:

- 회사 회의 시작 알림
- 이미 상태가 업데이트된 작업
- 완료된 작업 반복 알림
- 사용자가 방해금지로 설정한 시간대
- 개인 비공개 일정의 상세 변경
- 완료 또는 취소된 결제 요청의 반복 알림

## 변경 공유 정책

개인이 자신의 일정을 직접 관리하되, 업무에 영향을 주는 변경은 팀에 보인다.

즉시 공유할 변경:

- 작업 시작/종료 시간이 변경됨
- 작업이나 마일스톤이 드래그로 다른 날짜/시간으로 이동됨
- 담당자가 변경됨
- 상태가 문제발생, 홀드, 시간변경필요로 바뀜
- 프로젝트 마일스톤과 연결된 작업이 지연됨
- 작업이 삭제되거나 필요없음으로 바뀜
- 다른 프로젝트나 작업과 병합됨
- 결제 요청 상태가 완료/대기/취소로 변경됨
- 회의록에서 Action Task가 생성됨
- 회의록 Action 후보의 담당자 또는 마감일이 누락됨

조용히 기록만 할 변경:

- 설명 문구 수정
- 체크리스트 세부 항목 수정
- 비공개 개인 일정의 제목 변경
- 알림 설정 변경

화면 표시 방식:

- 팀 캘린더: 변경된 작업에 `수정됨` 배지를 일정 시간 표시한다.
- 팀 현황판: 최근 변경 내역 패널에 노출한다.
- 프로젝트 화면: 프로젝트 관련 변경만 상태 이력에 남긴다.
- 오늘 화면: 내가 관련된 변경만 보여준다.

팀이 지금 봐야 하는 항목만 따로 모은다.

- 응답 대기
- 문제발생
- 홀드
- 마감 임박
- 일정 충돌
- 담당자 없음

### 3. 일정 충돌 감지

작업 시간과 회사 일정이 겹치면 변경을 제안한다.

```text
황성현님은 13:00에 회의가 있습니다.
"상세페이지 QA" 시간을 변경할까요?
```

퇴근 전 또는 하루 끝에 미완료 작업을 정리한다.

- 오늘 완료한 작업
- 미완료 작업
- 내일로 이동할 작업
- 문제발생 작업

### 5. 계획 vs 실제 기록

부담스럽지 않은 수준으로 예정 시간과 실제 시간을 기록한다.

- 예정 시작/종료
- 실제 시작/완료
- 변경 사유
- 반복 지연 작업

## 현재 권장 방향

Que는 처음부터 대형 프로젝트 관리툴처럼 만들기보다, 8명 팀이 매일 실제로 열어볼 수 있는 가벼운 업무 운영 도구로 시작하는 것이 좋다.

우선순위는 다음과 같다.

1. 오늘 할 일과 상태 응답을 빠르게 처리한다.
2. 팀 전체의 문제발생/홀드를 빨리 드러낸다.
3. 회사 캘린더와 작업 일정을 한 화면에서 본다.
4. 자연어 입력은 보조 수단으로 제공한다.
5. 분석과 리포트는 사용 데이터가 쌓인 뒤 확장한다.
