# Claude Code 플랜 계획 프롬프트

아래 프롬프트는 개발 전에 Claude Code에게 구현 계획을 세우게 할 때 사용한다.

```text
Que 앱 개발 플랜을 세워줘. 아직 코드는 수정하지 말고, 먼저 레포를 읽고 실행 가능한 계획만 작성해.

반드시 읽을 자료:
- docs/que-product-plan.md
- preview/index.html
- preview/now.html
- preview/today.html
- preview/calendar.html
- preview/all-calendar.html
- preview/horizontal-calendar.html
- preview/team.html
- preview/heatmap.html
- preview/meeting-notes.html
- preview/action.html
- preview/project.html
- preview/payment.html
- preview/styles.css
- preview/app.js

전제:
- 알림과 설정 화면은 개발 후순위다. 프리뷰가 별도 제공된 뒤에 착수하므로 이번 플랜에서는 범위에서 제외하거나 마지막 Phase 이후로 미룬다.

계획에 반드시 포함할 것:
1. 현재 레포 상태 요약
2. 권장 기술 스택과 이유
3. shadcn/ui 설치 및 컴포넌트 추가 순서
4. App Shell, 라우팅, 레이아웃 구조
5. mock data와 타입 모델 설계
6. 메뉴별 구현 순서
7. 태블릿/FHD 대응 전략
8. 드래그 이동 구현 전략
9. 회의록 MD 업로드와 Action Task 생성 구현 전략
10. 권한/공개 범위의 MVP 처리 방식
11. 테스트와 검수 계획
12. 1차 MVP에서 제외할 항목

우선순위 기준:
- 태블릿에서 실제로 확인하고 수정하기 쉬운가
- FHD에서도 정보가 넓고 안정적으로 보이는가
- 상태 변경과 일정 이동이 빠른가
- 회의록 Action이 Task로 누락 없이 이어지는가
- 감시가 아니라 병목 공유로 보이는가

출력 형식:
- Phase 0: 프로젝트 세팅
- Phase 1: App Shell + 디자인 시스템
- Phase 2: 데이터 모델 + mock data
- Phase 3: 핵심 운영 화면
- Phase 4: 회의록/Action/Now 연결
- Phase 5: 프로젝트/결제/히트맵
- Phase 6: 검수와 안정화

각 Phase마다 다음을 써줘:
- 목표
- 구현 파일/폴더 예상
- 완료 조건
- 리스크
- 검증 방법

코드 수정은 하지 말고 계획만 작성해.
```

## 플랜 승인 후 실행 프롬프트

```text
방금 만든 Que 개발 플랜을 기준으로 Phase 0부터 구현해줘.

규칙:
- 한 Phase가 끝날 때마다 실행한 변경, 검증 결과, 다음 Phase 리스크를 보고해.
- shadcn/ui 컴포넌트를 우선 사용해.
- 태블릿 가로/세로와 FHD에서 깨지는 레이아웃이 없게 한다.
- 캘린더와 테이블은 내부 스크롤과 sticky header/label을 사용한다.
- mock data는 실제 API로 교체하기 쉬운 구조로 분리한다.
- 임시 스타일이라도 의미 없는 장식용 UI는 만들지 않는다.
```
