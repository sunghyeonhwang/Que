> 작성 2026-07-10. **C-3b 홈 정식 디자인(Figma)과 후속 구현의 To-be 명세**다.
> 기획의 유일한 정본은 `data/docs/que-product-plan.md`이며, 이 문서는 그중 직급별 홈 결정을 Figma와 구현에 필요한 수준으로 풀어 쓴다. 두 문서가 충돌하면 제품 기획서를 먼저 고친 뒤 이 문서를 동기화한다.

## 0. 목적과 설계 원칙

- **홈 = “오늘 무엇을 봐야 하는가”에 대한 역할별 답**이다. 감시 화면이 아니라 병목·충돌·요청을 빨리 드러내는 운영 화면이다.
- 대표·관리자·사원 홈은 서로 다른 제품이 아니라 같은 Home Dashboard의 `role` variant다.
- 같은 기능은 같은 화면 제목과 Figma 컴포넌트 이름을 쓴다. 역할 차이는 helper text, 데이터 스코프, 허용 액션으로 표현한다.
- 숫자보다 다음 행동이 먼저 보여야 한다. 첫 화면에서 `우선 확인`, `오늘 할 일`, `오늘 일정` 중 역할에 필요한 항목을 확인할 수 있어야 한다.
- 개인 이름이 붙는 부하·히트맵·성과 지표는 개인 평가가 아니라 업무 배분·병목 조정용임을 화면에서 명시한다.
- 역할 판정은 서버가 세션 사용자로만 한다. URL·클라이언트 입력으로 상위 화면을 열 수 없다.
- 히어로·장식 그래픽·카드 중첩을 만들지 않는다. 표·상태·액션을 먼저 보여준다.

### 역할 판정과 페이지 제목

| grade | 판정 | Figma 프레임 | 화면 제목 | helper text |
|---|---|---|---|---|
| `ceo` | rank=`대표` | `Home / CEO / Desktop` | **전사 현황** | 전사 진척과 위험, 결정이 필요한 항목을 확인합니다. |
| `manager` | rank=`관리` | `Home / Manager / Desktop` | **팀 운영** | 오늘 조정하거나 확인해야 할 병목·충돌·부하를 봅니다. |
| `staff` | 그 외(누락 포함) | `Home / Staff / Desktop` | **내 하루** | 오늘 내 작업과 일정, 확인할 요청을 봅니다. |

- 판정 함수는 core `gradeForUser(user)`를 사용한다. rank 문자열을 화면·데이터 계층에서 직접 비교하지 않는다.
- 쓰기 권한 축(`role: admin|member`)과 노출 축(`grade`)은 별개다. grade는 무엇이 보이는지, role과 core 권한 함수는 무엇을 고칠 수 있는지 정한다.

## 1. 공통 네이밍 계약

### 화면 표시명과 Figma 컴포넌트

| Figma 이름 | 화면 표시명 | 용도 |
|---|---|---|
| `Home/Header` | 역할별 페이지 제목 | 제목·helper text·클라이언트 기준 배지 |
| `Home/TodaySummary` | **오늘 요약** (대표 variant: **경영 AI 브리핑** · 관리자 variant: **팀관리 AI 브리핑**) | 짧은 요약. 사원=규칙 기반, **대표·관리자=AI 생성 허용(2026-07-10 통일 결정)** — 각자 스코프 데이터만 전송 |
| `Home/KPIGroup` | **핵심 현황** | 역할 스코프에 맞는 compact 지표 묶음 |
| `Home/KPICard` | 개별 KPI | 공통 카드의 role·tone variant |
| `Home/Priority` | **우선 확인** | 지금 응답·조정·결정할 항목 |
| `Home/PriorityItem` | 개별 우선 항목 | 상태·맥락·허용 액션이 있는 행 |
| `Home/CheckIn` | **작업 상태 확인** | 사원에게만 조건부로 나타나는 자동 체크인 응답 카드 |
| `Home/TodayTasks` | **오늘 할 일** | 로그인 사용자 본인의 오늘·기한초과 작업 |
| `Home/TodaySchedule` | **오늘 일정** | 로그인 사용자 본인의 오늘 작업·일정 |
| `Home/ProjectOverview` | **프로젝트 현황** | 대표·관리자의 프로젝트 건강도 |
| `Home/WorkflowTrend` | **업무 흐름** | 완료·신규·기한초과·적체 추이 |
| `Home/Workload` | **업무 부하** | 대표·관리자의 업무 배분용 사람·날짜 부하 |
| `Home/Pending` | **처리 대기** | 결제·회의록 Action 등 보조 대기 신호. `확인 필요`는 회의록 Action(needs_review)의 고유 용어라 카드명으로 재사용하지 않는다(2026-07-10 결정 — 중첩 명명 회피). Action 항목은 항상 풀네임 `확인 필요 Action`으로 쓴다 |
| `Home/SectionCard` | 공통 섹션 카드 | 제목·메타/액션·본문 계약 |

**네이밍 변경 범위는 홈에 국한하지 않는다(2026-07-10 결정)** — `상태 응답 대기`·`처리 대기` 등 이 계약의 용어는 팀 현황·성과·도움말을 포함한 **전 화면**에 적용한다(예: 현행 `응답 대기` 라벨 전면 교체).

역할마다 `ManagerKpiCard`, `StaffKpiCard`처럼 별도 이름의 컴포넌트를 만들지 않는다. `Home/KPICard`와 `Home/PriorityItem`에 `role=ceo|manager|staff`, `tone`, `state` variant를 둔다.

### 자동 체크인 용어

| 사용 위치 | 고정 명칭 |
|---|---|
| 기획·도메인·개발 | `자동 체크인` |
| Figma 컴포넌트 | `Home/CheckIn` |
| 사원 화면 카드 | `작업 상태 확인` |
| 대표·관리자 우선 항목 상태 | `상태 응답 대기` |

`체크인 요청`, `상태 체크`, `작업 체크인`을 화면 문구로 혼용하지 않는다.

## 2. 공통 프레임과 인터랙션

| 요소 | 목표 동작 |
|---|---|
| 헤더 좌측 | 역할별 제목·helper text·클라이언트 필터 배지(`○○ 기준`) |
| 헤더 우측 | 작업 추가. 팀원 아바타 스택은 대표·관리자만 표시하고 사원은 현재 사용자 맥락만 유지 |
| 우하단 | FAB 작업 추가. 상단 버튼과 같은 자연어 모달(해석→확인 카드→등록) |
| URL 상태 | 월 선택이 필요한 variant만 `?hm=1..12` 사용. 잘못된 값은 현재월 폴백 |
| 클라이언트 스코프 | 작업·일정·KPI·프로젝트·업무 흐름·부하는 현재 클라이언트 필터를 존중 |
| 필터 비적용 | 회의록 Action·결제·개인 알림 뱃지는 클라이언트 필터의 영향을 받지 않음 |

### 공통 순서

```text
Home/Header
Home/TodaySummary
Home/KPIGroup
Home/Priority
Home/TodayTasks + Home/TodaySchedule
Home/ProjectOverview        # 대표·관리자
Home/WorkflowTrend
Home/Workload               # 대표·관리자
Home/Pending                # 대표·관리자
```

- 역할에 필요 없는 섹션은 빈 카드로 남기지 않고 제외한다.
- `Home/Priority`는 홈에 최대 5건만 표시하고 `전체 보기`를 제공한다.
- `Home/TodayTasks`, `Home/TodaySchedule`, `Home/Priority`는 높이 상한 뒤 내부 세로 스크롤을 허용한다. 페이지 가로 스크롤은 금지한다.
- 표·히트맵·날짜 그리드는 카드 내부 가로 스크롤과 sticky header/label을 사용한다.
- 동일 데이터의 상단바 알림·홈·상세 화면 간 수치와 필터 기준을 문서로 명시한다. 같은 모집단이 아니면 같은 수치처럼 표현하지 않는다.

## 3. 사원(`staff`) 홈

**한 줄 정의**: 출근해서 이 화면 하나로 “오늘 무엇을 하고, 무엇에 응답해야 하는가”를 확인한다. 팀 지표·타인 데이터는 없다.

### 섹션 계약

| 순서 | 공통 섹션 | 구성 | 데이터·규칙 |
|---|---|---|---|
| 1 | `Home/TodaySummary` | 오늘 할 일·마감·응답 필요 수를 1~2문장으로 요약 | 규칙 기반. AI 생성 버튼·경영 브리핑 표현 없음 |
| 2 | `Home/KPIGroup` | 오늘 할 일·진행 중·이번 주 완료·기한 초과 compact 스트립 | **본인 작업만** 집계. 점수·등수·타인 대비 없음 |
| 3 | `Home/Priority` | 작업 상태 확인·도움 요청·확인 필요 Action·본인 기한초과 등 최대 5건 | **본인에게 관련된 신호만**. `Home/CheckIn`이 있으면 첫 항목 |
| 4 | `Home/TodayTasks` | 제목·거래처/프로젝트·상태·마감·완료 액션 | 본인 작업만. 완료 변경은 기존 core mutation과 ChangeLog 경유 |
| 4 | `Home/TodaySchedule` | 시간·제목·종류·관련 프로젝트 | 본인 일정. 외부 회사 일정은 읽기 전용 |
| 5 | `Home/WorkflowTrend` | 완료·신규·기한초과 추이 + 본인 한 행 월간 작업 기록 | 개인 평가가 아니라 본인 적체·흐름 확인용 |

### 사원 우선 확인 데이터

- 현행 `getAlerts`는 모든 문제·기한초과·결제 신호를 모으므로 그대로 “본인 대상”이라고 간주하지 않는다.
- 후속 구현은 viewer-scoped selector를 제공해야 한다. 사원에게는 다음만 포함한다.
  - 본인이 담당한 문제·홀드·기한초과 작업
  - 본인에게 온 도움 요청·상태 확인 요청
  - 본인이 확인할 수 있고 담당/마감 보완이 필요한 Action
  - 본인이 요청자·처리 대상인 결제 신호
  - 본인의 응답 대기 자동 체크인
- **상단바 벨·알림 센터(`/notifications`)도 같은 viewer-scoped selector를 사용한다(2026-07-10 결정: 벨 개인화).** 벨·알림 센터·사원 `우선 확인`은 같은 모집단이라 수치가 항상 일치해야 한다. 전역 운영 신호(팀 병목 전체 등)는 관리자·대표의 `우선 확인`과 팀 현황 화면이 커버한다.

### 자동 체크인

- Que에서 생성된 작업이 시작되거나 설정된 확인 시점이 되었을 때만 생성한다.
- 회의·휴가·외부 일정에는 묻지 않고, 이미 상태가 업데이트된 작업에는 다시 묻지 않는다.
- 응답 대기가 있으면 `Home/Priority`의 첫 항목에 `Home/CheckIn`을 표시한다. 없으면 카드 전체를 숨긴다.
- 질문: `‘{작업명}’ 작업은 어떻게 진행되고 있나요?`
- 선택지: 작업 중 · 완료 · 시간 변경 필요 · 문제 발생 · 필요 없어짐 · 다른 프로젝트와 병합 · 나중에.
- 문제발생·홀드·병합처럼 추가 정보가 필요한 응답은 후속 입력으로 연결한다. 문제·홀드는 사유·다음 액션·도움 대상·재확인 시간을 함께 받는다.
- 여러 건이면 `1 / N` 진행 상태를 표시하되 한 번에 한 작업만 묻는다.

### 빈 상태

- 오늘 할 일 0건: `오늘 마감이거나 기한이 지난 작업이 없습니다.`
- 우선 확인 0건: `지금 확인할 요청이 없습니다.` 또는 섹션 축소.
- 오늘 일정 0건: `오늘 등록된 일정이 없습니다.`
- 자동 체크인 0건: 빈 카드를 만들지 않음.

## 4. 관리자(`manager`) 홈

**한 줄 정의**: 아침에 열면 오늘 조정하거나 확인할 팀 병목·충돌·부하를 30초 안에 파악하고, 본인 실행 항목도 첫 흐름에서 놓치지 않는다.

### 섹션 계약

| 순서 | 공통 섹션 | 구성 | 데이터·규칙 |
|---|---|---|---|
| 1 | `Home/TodaySummary` | **팀관리 AI 브리핑** — 병목·충돌·부하 요약 + `AI 브리핑 생성` 버튼 | **AI 생성 승인됨(2026-07-10, 대표와 통일)**. E-10 재사용(온디맨드), 전송 범위=**관리자 스코프**(팀 작업·부하 — 대표 개인 데이터 제외), `전송 데이터 보기` 고지 필수, 미생성 시 규칙 기반 폴백 |
| 2 | `Home/KPIGroup` | 활성 프로젝트·열린 작업·현재 막힘·마감 임박·일정 충돌·상태 응답 대기 | 숫자만 두지 않고 관련 필터 화면으로 연결 |
| 3 | `Home/Priority` | 문제발생·홀드·도움 요청·일정 충돌·상태 응답 대기 최대 5건 | 문제→홀드→도움 요청→충돌→응답 대기 순. 대표 담당 작업의 문제·홀드 포함 |
| 4 | `Home/TodayTasks` | 관리자 본인의 오늘 할 일 | 타인 작업 직접 수정 UI 없음 |
| 4 | `Home/TodaySchedule` | 관리자 본인의 오늘 일정 | 개인 실행 맥락을 팀 목록 뒤로 보내지 않음 |
| 5 | `Home/ProjectOverview` | 거래처/프로젝트·진행률·막힘·마감 위험·상태 | 위험도가 높은 활성 프로젝트 최대 5개. 클라이언트 필터 적용 |
| 6 | `Home/WorkflowTrend` | 4/8/12주 **라인 차트 3선**(신규·완료·기한초과) + 우상단 **순증 배지**(예: "이번 주 순증 −2 (적체 감소)") | **B안 확정(2026-07-10)** — 순증감은 파생값·음수 가능이라 선에 안 섞고 배지로. 성과 화면 라인 차트 컴포넌트 재사용. X축 라벨은 주차(W1)가 아니라 **날짜 구간(MM.DD-MM.DD)**. 개인별 성과 비교 없음 |
| 7 | `Home/Workload` | 이름·예상/가용 시간·열린 작업·마감 임박·막힘·영향 프로젝트 | **대표 제외**. 고정 사용자 순서, 순위표 아님 |
| 8 | `Home/Pending` | 확인 필요 Action·결제 대기·장기 상태 응답 대기 | compact summary. 각 상세 화면으로 연결 |

### 우선 확인 액션

- 홈에서 타인의 작업 상태를 직접 바꾸지 않는다.
- 허용 액션은 상세 보기·일정 보기·댓글·도움 요청·상태 확인 요청이다.
- 기존 허용 mutation이 없는 액션은 Figma에서 작동하는 버튼처럼 확정하지 않고 딥링크로 표현한다.
- 문제·홀드 항목은 작업명·거래처/프로젝트·담당자·사유·도움 대상·다음 확인 시각을 우선 노출한다.
- `상태 응답 대기`는 자동 체크인의 관리자 표현이다. 자동 체크인을 별도 메인 섹션으로 만들지 않는다.

## 5. 대표(`ceo`) 홈

**한 줄 정의**: 전사 위험과 흐름, 결정이 필요한 항목을 한 화면에서 훑고 본인 실행 항목도 같은 홈에서 확인한다.

## 6. 데이터 스코프와 권한 방어

1. **분기**: `getGradeHomeData(user, …)`가 세션 user에서 grade를 판정한다. URL·파라미터로 상위 홈을 요청할 수 없다.
2. **데이터 게이트**: 관리자·대표 전용 번들은 role 게이트와 viewer 재유도를 통과해야 한다.
3. **타입 분리**: staff 홈에는 타인 부하·전사 프로젝트·클라이언트 집계 prop을 전달하지 않는다.
4. **개인 알림**: 사원 `Home/Priority`는 viewer 관련 신호만 반환하는 selector를 사용한다.
5. **편집 권한**: 본인 작업만 수정한다. 타인 작업은 댓글·도움 요청·상태 확인 요청만 제공한다.
6. **비공개·PII**: 타인의 비공개 일정 상세와 PII를 홈에 노출하지 않는다.
7. **상태 변경**: 문제·홀드·일정·담당자 등 업무 영향 변경은 core mutation과 ChangeLog를 거친다.

## 7. 반응형·접근성

| 해상도 | 레이아웃 |
|---|---|
| FHD 1920 | 풀 사이드바 236px. 역할별 2단 그리드와 KPI 가로 스트립 유지 |
| 1366×768 | 풀 사이드바. 핵심 항목이 첫 화면에 보이도록 세로 높이와 목록 상한 적용 |
| 태블릿 가로 1024×768 | 72px Sidebar Rail. KPI 2×2 또는 3×2, 표·히트맵 내부 스크롤 |
| 태블릿 세로 768×1024 | 사이드바 대신 상단 `MobileNav` Sheet. 모든 주요 섹션 단일 컬럼 |

- 버튼·행·완료 토글·선택지는 최소 40px, 가능하면 44px 이상이다.
- 색상만으로 상태를 구분하지 않는다. 상태 텍스트·아이콘·수치를 함께 표시한다.
- icon-only 버튼은 `aria-label`과 Tooltip을 갖는다.
- 표 header/cell 의미, 키보드 포커스, Dialog/Sheet focus trap을 유지한다.
- 125% 확대에서 핵심 조작이 유지되어야 한다.
- 태블릿 세로에서 표·히트맵 외의 페이지 가로 스크롤을 만들지 않는다.

## 8. Figma 작업 규칙

- 대표 레퍼런스 node 76:18의 App Shell·다크 테마·밀도·간격·카드 스타일을 공통 스타일 마스터로 사용한다.
- 프레임 이름은 `Home / {Role} / {Viewport}` 형식을 사용한다.
  - `Home / CEO / Desktop`
  - `Home / Manager / Desktop`
  - `Home / Staff / Desktop`
  - `Home / {Role} / Tablet Landscape`
  - `Home / {Role} / Tablet Portrait`
- 공통 컴포넌트는 §1의 `Home/*` 이름을 사용하고 role variant로 분기한다.
- shadcn 글로벌 theme token을 브랜드 색으로 덮어쓰지 않는다.
- 상태 의미는 green=진행/완료, blue=예정/정보, amber=주의/대기, red=문제/취소, violet=회의록/응답대기를 유지한다.
- 카드 안에 카드를 중첩하지 않는다. 보조 지표는 같은 섹션의 panel/row로 표현한다.
- 라이트·다크를 별도 컴포넌트로 복제하지 않고 같은 token/variable mode로 만든다.

## 9. 현행 대비 구현 필요사항

이 절은 To-be 명세와 2026-07-10 현행 코드의 차이다. Figma 완료가 곧 구현 완료를 뜻하지 않는다.

- 공통 네이밍과 역할별 페이지 제목 변경 — **전 화면 범위**(`응답 대기`→`상태 응답 대기` 등 팀 현황·성과·도움말 포함).
- `Home/KPIGroup` 역할별 지표 신규 산출 — 사원 `오늘 할 일`·`이번 주 완료`, 관리자 `일정 충돌`·`상태 응답 대기` 수 등은 현행 성과 KPI에 없는 파생 지표.
- **상단바 벨·알림 센터를 viewer-scoped selector로 전환**(벨 개인화 결정) — C-3a 알림 센터의 전역 문구("지금 주의가 필요한 항목")·도움말 s1·"벨과 같은 소스" 주석 동반 갱신.
- 규칙 기반 `Home/TodaySummary` 신규 조립 + **대표 `경영 AI 브리핑`·관리자 `팀관리 AI 브리핑`**(E-10 Gemini 재사용, 각자 스코프 전송, 전송 데이터 고지, 폴백=규칙 기반). 사원은 규칙 기반만.
- 사원 viewer-scoped `Home/Priority` 데이터 selector. 현행 전역 `getAlerts` 직접 재사용 금지.
- 사원 조건부 `Home/CheckIn` 연결.
- 홈 `오늘 할 일` 완료 액션은 현재 상세 링크만 있으므로 core mutation을 재사용해 별도 구현.
- 관리자 `Home/Priority` 상위 5건 정렬·전체보기, 프로젝트 현황, 업무 흐름 추가.
- 대표 홈을 공통 섹션 계약과 순서에 맞게 재구성.
- 768px 내비게이션 명세를 Sidebar Rail이 아닌 `MobileNav` Sheet에 맞춤.

## 10. 수용 기준

- 세 역할의 동일 기능이 같은 `Home/*` 이름과 Figma component instance를 사용한다.
- 사원 홈에서 타인 작업·팀 부하·전사 프로젝트 집계가 노출되지 않는다.
- 자동 체크인은 사원에게 조건부 `작업 상태 확인`, 관리자·대표에게 `상태 응답 대기`로 표현된다.
- 관리자·대표의 부하 화면은 개인 평가·순위로 읽히지 않는다.
- 상단바 벨·알림 센터·사원 `우선 확인`이 같은 viewer-scoped 모집단을 쓰고 수치가 일치한다.
- 클라이언트 필터 적용/비적용 데이터가 §2 계약과 일치한다.
- 우선 확인은 최대 5건이며, 전체보기 또는 상세 딥링크가 있다.
- 문제·홀드 상태는 사유·다음 액션·도움 대상·재확인 시간 계약을 지킨다.
- 1920×1080, 1366×768, 1024×768, 768×1024에서 핵심 정보·터치·내부 스크롤을 검수한다.
- 구현 시 lint·typecheck·core test·build·4해상도 브라우저 검증 결과를 남긴다.
