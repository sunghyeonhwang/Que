## 작성 규칙

- **판정은 정직하게.** 문제가 있으면 Ship이라 하지 않는다. Blocker가 하나라도 있으면 Block, High만 있으면 Fix-then-ship.
- **각 발견은 근거를 단다.** `파일:줄` 또는 스샷 번호. 근거 없으면 리포트에 올리지 않는다.
- **"왜 중요한가"는 Que 맥락으로.** 일반론("사용성 향상")이 아니라 "이게 병목을 가린다/규칙을 어기게 한다/이 해상도에서 못 누른다"처럼 운영 임팩트로 쓴다.
- **수정안은 실행 가능하게.** 방향 + 난이도. 코드는 사용자가 요청할 때만 작성한다.
- 발견이 없으면 "발견 없음 — 통과" 표를 남기고 왜 통과인지 근거를 든다. 억지로 채우지 않는다.
