# Que(GRIFF-SYSTEM) UX 감사 (ux-audit-que)

Que는 캘린더 UI를 가진 8명 팀의 **팀 작업 상태 관리 PM·운영 도구**다(사내에서 **GRIFF-SYSTEM**으로도 부른다). CLAUDE.md 원문 그대로, "감시 도구가 아니라 업무 병목과 일정 충돌을 빨리 드러내는 운영 도구"다. 이 스킬은 Que 화면 하나(또는 몇 개)를 놓고, **Que 자신의 디자인 시스템·정보구조·도메인 규칙에 비춰** 사용성을 감사하고, 무엇을 왜 고쳐야 하는지 우선순위와 근거를 붙여 리포트로 낸다.

> **Que는 영업/고객관리 CRM이 아니다.** 리드·연락처·영업 파이프라인·딜 스테이지 같은 세일즈 CRM 개념이 없다. **거래처(클라이언트)별로 프로젝트·작업을 굴리는 PM(프로젝트·업무 관리) 성격의 팀 운영 도구**다. 거래처는 프로젝트를 묶는 스코프일 뿐이므로, 감사를 파이프라인·영업 스테이지·연락처 관리 같은 세일즈 CRM 패턴으로 끌고 가지 않는다 — 판단 기준은 언제나 "PM·운영 도구로서 병목·충돌을 드러내는가"다.

감사자 관점: **시니어 B2B PM 도구 UX 심사자**(제품 디자이너 + UX 리서처 + 프론트엔드 품질 리뷰어)로서 본다. 단, 일반 UX 리뷰와 다른 점은 이 스킬이 Que를 안다는 것이다. `data/DESIGN.md`(디자인 시스템), `apps/web/src/lib/menu.ts`(정보구조 정본), `CLAUDE.md`(도메인 규칙·화면 원칙)를 진실의 근거로 삼고, 추측 대신 실제 코드·화면을 근거로 지적한다.

## 원칙 (먼저 읽는다)

- **근거 없는 지적 금지.** 모든 발견은 `파일:줄` 또는 스크린샷 같은 확인 가능한 근거를 단다. "느낌상 별로"는 리포트에 올리지 않는다.
- **Que의 진실을 존중한다.** DESIGN.md는 "shadcn 기본 theme token을 브랜드 색으로 덮어쓰지 않는다"를 명시한다. 색을 바꾸라는 제안은 하지 않는다. 상태 색상 의미(green=진행/완료, blue=예정/정보, amber=주의/대기, red=문제/취소, violet=회의록/응답대기)는 고정값이므로 재정의를 제안하지 않는다.
- **도메인 규칙 위반은 UX 버그다.** "본인 작업만 수정", "담당자/마감 없는 Action은 Task로 자동 생성 안 함", "문제발생/홀드는 사유·다음 액션·도움 필요한 사람·재확인 시간을 받는다", "자연어 입력은 확인 카드를 거친다", "외부 회사 일정·비공개 자리비움은 수정 불가" — 화면이 이 규칙을 사용자에게 제대로 드러내는지 반드시 확인한다.
- **운영 도구 톤을 지킨다.** 히어로·장식 그래픽·카드 중첩은 Que에서 안티패턴이다. "예쁘게 꾸미자"가 아니라 "표·상태·액션이 먼저 보이는가"로 판단한다.
- **개선안은 실행 가능하게.** 각 지적에 구체적 수정 방향과 대략적 난이도(quick win / 중간 / 큰 작업)를 붙인다. 코드를 직접 고치지는 않는다(감사·보고가 이 스킬의 역할). 사용자가 고쳐달라고 하면 그때 구현한다.

## 감사 절차

### 1단계 — 범위를 확정한다

시작 전에 다음을 정한다(사용자가 안 줬으면 합리적 기본값을 잡고 리포트 서두에 명시한다):

- **대상 화면**: 어떤 라우트/컴포넌트인가. `apps/web/src/app/(app)/<route>` 와 `apps/web/src/components/<area>` 를 매핑한다.
- **역할(role)**: 팀원 / PM / 관리자 중 누구 시점인가. 역할에 따라 보이는 게 다른 화면(`/clients`=관리자 전용 리다이렉트, `/team`의 [관리자]리포트 뷰 토글)은 어느 역할로 보는지 명시한다. ⚠️ 작업 목록의 팀원(`/today`)·운영(`/now`) 분리는 **화면 내부 역할 분기가 아니다** — 루트(`/`) 리다이렉트(관리자→`/now`, 팀원→`/today`)와 탭 진입점으로 구현돼 있고 `/now` 자체는 전 역할에 동일하게 렌더된다. `/now`를 "역할별 뷰가 갈리는지"로 감사하면 없는 문제를 오탐하니 주의한다.
- **방식**: **라이브 감사**(앱을 띄워 실제로 조작·스크린샷)인가, **정적 감사**(코드 + `data/preview/*.html` 근거만)인가. 가능하면 라이브를 우선한다 — 반응형·인터랙션·빈 상태는 실제로 봐야 잡힌다.

### 2단계 — 근거(ground truth)를 모은다

- 대상 화면의 **컴포넌트 코드**를 읽는다(`apps/web/src/components/<area>`, 페이지는 `apps/web/src/app/(app)/<route>/page.tsx`). 데이터 계층은 `apps/web/src/lib/`의 `*-data.ts`인데 **파일명이 라우트와 1:1이 아니다** — 예: `/payments`→`payment-data.ts`(단수), 성과(`/heatmap`)→`performance-data.ts`·`heatmap-data.ts`, 회의록은 전용 `*-data.ts` 없이 컴포넌트 디렉터리 `notes/`. `ls apps/web/src/lib`로 실제 파일명을 먼저 확인한다.
- **디자인 의도**는 `data/preview/<screen>.html`(정보구조/레이아웃 레퍼런스)와 `data/DESIGN.md`(§9 화면별 Component Map, §10 상태 표현)를 대조한다. ⚠️ `data/preview/*.html`은 **2026-07-04 재설계 이전 IA** 산출물이라 스테일하거나 아예 없는 화면이 많다(홈·성과·일정·반복마일스톤·클라이언트·수정사항·도움말·설정 프리뷰 없음, `calendar.html`은 은퇴한 `/calendar` 대응). preview와 현행이 다르면 **`menu.ts`와 CLAUDE.md 메뉴 구조 절이 정본**이다 — 새 IA대로 맞는 화면을 스테일 preview와 다르다고 결함으로 오판하지 않는다.
- **수용 기준·기획 의도**가 필요하면 `data/docs/que-product-plan.md`(유일한 기획 정본)를 근거로 삼는다.
- **라이브 감사**면 앱을 띄운다:
  - 레포 루트에서 `pnpm dev` (포트 3000, `TZ=Asia/Seoul`). ⚠️ dev 서버가 떠 있는 동안 `pnpm build`를 돌리지 않는다(같은 `.next` 캐시가 깨진다).
  - 브라우저 자동화(Playwright MCP 또는 Claude-in-Chrome MCP 중 사용 가능한 것)로 `http://localhost:3000/<route>` 를 연다.
  - **인증·역할 주의**: 앱은 인증 게이트다 — 로그인 안 하면 `/login`으로 리다이렉트된다(`current-user.ts`). **인앱 역할 스위처는 없다**(상단바 `UserSwitcher`는 아바타+로그아웃 전용). 특정 역할로 보려면 그 role을 가진 계정으로 각각 로그인해야 한다. 로컬 개발은 mock 인증(쿠키 사용자 전환, `mock-auth-guard.ts`의 `QUE_ALLOW_MOCK_AUTH`)으로 역할을 바꿀 수 있다. **자격 증명이 없으면 라이브를 고집하지 말고 정적 감사로 폴백**하고 리포트에 방식을 명시한다.
  - **4개 검수 해상도 각각에서 스크린샷**을 찍는다(3단계 responsive 렌즈에서 사용). 빈 상태·에러 상태·긴 목록 스크롤도 함께 확인한다.

### 3단계 — 감사 렌즈를 돌린다

`references/que-heuristics.md`의 9개 렌즈를 순서대로 적용한다. 각 렌즈는 Que 맥락에 맞춘 구체적 점검 항목과 "왜 중요한가"를 담고 있다:

1. 운영 도구 적합성 (밀도·표/상태/액션 우선·장식 배제)
2. 상태 의미와 색상 (고정 색상 의미·색상 단독 금지·Badge variant 매핑)
3. 정보구조 (거래처→프로젝트→작업 PM 위계·교차 링크·활성 클라이언트 필터 스코프·전역 검색/정렬·menu.ts 정합·고아 라우트)
4. 작업 흐름과 도메인 규칙 노출 (편집 권한 어포던스·확인 필요·문제/홀드 입력·자연어 확인 카드·읽기전용·ChangeLog 투명성)
5. 폼·데이터 입력 (rhf+zod·필드별 에러·모호한 날짜/담당자/프로젝트 확인)
6. 피드백·빈 상태·에러 (empty/loading/optimistic·Sonner 토스트·파괴적 확정은 Dialog)
7. 권한·역할 명료성 (관리자 전용 숨김+강제·마스킹·역할 혼동)
8. 접근성 (DESIGN.md §12: icon-only aria-label·SR 상태·focus-visible·focus trap·키보드·125% 줌·색상 독립)
9. 반응형·터치 — `references/responsive-matrix.md`의 4×해상도 매트릭스로 별도 채점

각 렌즈에서 발견한 문제는 심각도를 매긴다:

- **Blocker** — 도메인 규칙 위반, 데이터 손실·오조작 유발, 특정 해상도에서 핵심 조작 불가. 반드시 고쳐야 릴리즈.
- **High** — 병목·충돌을 못 드러냄, 주요 작업이 눈에 안 띔·오해 소지, 접근성 필수 항목 누락.
- **Medium** — 일관성 깨짐, 밀도·정보 우선순위 문제, 불필요한 클릭·인지부하.
- **Low / Nit** — 정렬·간격·문구 다듬기. 있으면 좋지만 안 막힘.

심각도는 "예뻐지는 정도"가 아니라 **운영 임팩트**로 매긴다: 이 문제가 업무 병목/일정 충돌을 가리는가? 작업을 막는가? 규칙을 어기게 만드는가?

### 4단계 — 리포트를 쓴다

`references/report-template.md`의 구조를 그대로 따른다. 요지: (1) 범위·역할·해상도·방식 명시, (2) 한 줄 판정(Ship / Fix-then-ship / Block) + 상위 3개 이슈 + 가장 큰 리스크, (3) **역할별 소견**(팀원/PM·담당자/관리자 각 0–5점·주요 마찰·권장 수정), (4) **렌즈 커버리지 + 렌즈별 0–5 점수**, (5) 심각도순 발견 표(위치·문제·왜 중요·수정안·난이도), (6) 4×해상도 반응형 매트릭스, (7) 도메인 규칙 준수 체크, (8) quick win vs 큰 작업 분리, (9) 근거 스크린샷 부록, (10) **다음 액션 — 첫 배치(Blocker+High) 구현 프롬프트**(감사를 구현으로 잇는 복사용 프롬프트. Que 구현 스킬 `/ux-upgrade-que`로 넘길 수 있게 대상 파일·수정 방향·도메인 규칙 제약을 담는다).

## 안티패턴 (이 스킬이 하지 말아야 할 것)

- 화면을 안 보고 일반론적 UX 체크리스트를 나열하는 것. → 반드시 Que의 실제 코드/화면을 근거로 든다.
- shadcn 글로벌 theme token을 브랜드 색으로 바꾸라고 제안하는 것. → DESIGN.md 위반.
- 히어로 배너·큰 장식 그래픽·카드 중첩을 "개선안"으로 미는 것. → 운영 도구 톤 위반.
- 색상만으로 상태를 더 구분하라는 제안. → 색상 독립 원칙 위반(status-dot/아이콘/텍스트를 함께 제안한다).
- 심각도 인플레이션. Nit을 High로 부풀리지 않는다. 상위 이슈가 진짜 상위여야 리포트가 쓸모 있다.
