# Que — 데일리 스탠드업 + OKR 초기 기획 (2026-07-11)

## Context — 왜 만드는가

Que는 전 구성원의 작업 상태(어제/오늘/막힘)를 이미 데이터로 알고 있다. 이를 활용해 **매일 오전 10시 데일리 스탠드업**을 온라인(비동기)으로 돌리고, 그 위에 **OKR(분기 목표 + 월 핵심결과)** 개념을 얹어 "오늘 하는 일이 어떤 목표에 기여하는가"를 드러낸다. 월요일 베타 시작의 운영 리듬(마일스톤 회의 → 데일리 스탠드업)을 도구가 받치는 것이 목적. **이번 산출물은 초기 기획이며 코드 구현은 하지 않는다.**

사용자 확정 방향: ①스탠드업+OKR 통합 기획(구현은 단계 분리) ②비동기 체크인 + AI(LLM) 진행 보조 ③OKR 주기 = 분기 Objective + 월 KR ④독립 메뉴 신설 ⑤회의 체계에 **마일스톤 회의** 추가.

### ① 월요일 주간 통합 회의 (Top-down 층 · 주 1회 — 2026-07-11 재편: 주간회의에 스탠드업·마일스톤 통합)

월요일은 회의를 쪼개지 않는다. **주간 전체 회의 하나**에 스탠드업(팀 라운드)과 마일스톤 안건이 섹션으로 흡수된다 — 별도의 월요일 10시 스탠드업·11시 마일스톤 회의는 없다(스탠드업은 화~금 10시만).

- **참석**: 전원 8인. **소요 목표 35~40분.**
- **섹션 구성(AI 아젠다 자동 생성 → AI 라이브 진행 — 상세는 §1-f)**:
  ⑴ 지난주 요약 ⑵ 이번 주 조망(금요일 프리뷰 §1-d 승계 + 부하 재배분) ⑶ **마일스톤 안건**(간트 페이지 §1-b — 위험만 보기로 압축 → 기한 조정 드래그·위험 판정·신규 등록은 LLM 채팅. 세부 작업 배분은 데일리 몫) ⑷ 결정 필요(Action 담당 지정·결제 승인·응답 없는 도움 요청) ⑸ **팀 라운드 = 월요일 스탠드업 대체**(AI가 막힘 우선 호명, 한마디는 standup_entries로 저장 — 월요일 체크인 기록 유지)
- **산출물**: 회의록 1건(kind=weekly, 마일스톤 결정 포함) + Action→Task 연결 + 마일스톤 ChangeLog.
- **Que 기획 요소**: 회의록 **kind: `milestone` | `weekly` | `general`** — weekly=월요일 통합 회의, milestone=주중 수시 마일스톤 처리(§1-c·§1-e). 데일리 보드에서 "이번 주 회의 결정" 참조 링크.

### ①-b 간트 페이지 — `gant.griff.co.kr` (2026-07-11 확정 · 주간 통합 회의 섹션 ⑶ 전용 화면)

- **인증(확정)**: **관리자·대표 계정만 로그인** 가능(회의실 화면 용도). 익명 조회 없음 — 일반 팀원 조회는 que.griff.co.kr의 기존 간트로 충분.
- **마일스톤 드래그 이동**(신규 개발, 중소형): 칩을 날짜 컬럼 스냅으로 끌어 기한 변경 — 기존 updateMilestoneAction 재사용, ChangeLog via=web.
- **전체화면**: 기존 Fullscreen 버튼 재사용. **스크롤**: 기존 ◀7일·오늘·▶7일 버튼 + 가로 스크롤.
- **위험만 보기 토글**: 주의·지연 마일스톤만 필터 — 회의 안건 자동화.
- **오늘 조정 요약 → 회의록 초안**: 회의 중 변경된 마일스톤 N건을 하단에 모아 표시, 회의 종료 시 그대로 회의록(kind=milestone) 초안으로 넘기기 — §1 회의 종류와 연결.

### ①-c 마일스톤 안건 AI 원격 진행 (2026-07-11 추가 · 주중 수시 처리)

월요일 통합 회의를 기다릴 수 없거나 모이지 못할 때, **AI가 사회자가 되어 안건을 하나씩 제시하고 답이 오면 다음으로 넘어가는** 비동기 진행 시나리오.

- **안건 큐 자동 생성**: 회의 개시(관리자 버튼 또는 정시) 시 AI가 안건을 자동 수집 — ⑴위험(주의/지연) 마일스톤 ⑵이번 주 마감 임박 마일스톤 ⑶지난 회의 이월 안건. 각 안건에 맥락 요약(관련 작업 진행률·막힘)을 붙인다.
- **1안건 1질문**: AI가 안건을 담당자에게 제시(Que 화면 + Slack DM 버튼 병행 — 기존 체크인 버튼 DM 인프라 재사용). 답변 옵션은 결정형: **기한 유지 / 기한 연기(새 날짜) / 위험 상태 변경 / 보류(사유)**. 답하면 즉시 반영(updateMilestoneAction, ChangeLog via=web)되고 다음 안건으로 넘어간다.
- **진행 화면**: /daily 또는 간트 페이지에서 "회의 진행 중 — 안건 3/7, 대기: ○○님" 상태 표시. 누구든 전체 큐와 결정 현황을 볼 수 있다.
- **종료·요약**: 전 안건 소진(또는 컷오프) 시 AI가 결정 요약을 생성해 **회의록(kind=milestone) 초안으로 저장** + Slack 팀채널 게시. 미응답 안건은 "다음 회의 이월"로 표시하고 담당자에게 재촉 1회.
- **권한**: 안건 응답은 해당 마일스톤 담당자·관리자만(기존 canManage 재사용). 개시·종료는 관리자.
- 로드맵 위치: **Phase 2.5**(스탠드업 AI 진행(Phase 2)의 안건 큐·버튼 응답·요약 패턴을 마일스톤 회의에 재적용 — 인프라 공유로 규모 중소형).

월요일 주간 통합 회의를 짧게 만드는 사전 예보. **AI가 금요일 오후(16:00 KST 권장) 다음 주 예보를 생성해 팀채널 게시.**

- **내용 3블록**: ⑴ 다음 주 마감 마일스톤(위험 상태·관련 작업 진행률) ⑵ 부하 쏠림 예상(부하 표 데이터 — 특정 인원 과부하 경고) ⑶ 선행 지연 위험(간트 선행 관계 기반, 기존 computeGanttRisk 재사용).

### ①-e 긴급 결정 — 자동 결정 워크플로 (2026-07-11 추가 · 2026-07-11 명확화: 회의가 아니다)

> **명확화(구 "위기 소집")**: 이것은 회의(모임)가 아니라 **응답을 요구하는 결정 절차**다. 모이지 않는다.
> 기존 알림과의 차이 — 알림(마감 임박·문제발생)은 **읽으면 끝**이지만, 긴급 결정은 **4옵션 중 하나를 답해야 종결**되고 답할 때까지 시스템이 추적한다.

- **주관**: AI(크론)가 감지·발송·추적을 전담한다. 사람은 열지 않는다 — 열리는 조건은 아래 트리거뿐.
- **트리거(3종)**: ⑴ 마일스톤이 '지연'으로 전환 ⑵ 선행 작업 지연으로 마일스톤 위험 감지(computeGanttRisk) ⑶ 마감 D-2인데 관련 작업 완료율 임계(기본 50%) 미만.
- **수신자**: 마일스톤 담당자 + 해당 프로젝트 PM, 딱 2명이 기본(관련 작업 담당자는 참조). 대표는 수신자가 아니라 **에스컬레이션 대상**.
- **흐름**: AI가 결정 카드 1장(맥락 요약 + 옵션: **기한 유지 / 연기(새 날짜) / 작업 재배분 / 보류(사유)**)을 DM → 누구든 한 명이 결정하면 즉시 반영(ChangeLog) + 팀채널에 결정 공유 → **종결**. 코멘트는 결정 전 협의용으로 자유.
- **미응답 에스컬레이션**: 2시간 미응답 → 재촉 1회 → 4시간 미응답 → 관리자(PM 외 1인)·대표에게 에스컬레이션 카드. 그래도 미응답이면 다음 날 주간/데일리 안건으로 이월 표시.
- **남발 방지**: 같은 마일스톤 24h 내 재발송 금지(dedup `crisis:<milestoneId>:<날짜>`), 하루 최대 3건 — 카드 피로가 곧 무시로 이어지므로 상한이 본질.
- 로드맵: **Phase 2.5에 포함**(§1-c 결정 카드·응답 인프라 공유 + 트리거 감지만 추가).

### ①-f 주간 통합 회의 — AI 아젠다·진행·회의록 상세 (§1의 실행 설계)

§1 월요일 주간 통합 회의에 AI를 세 지점으로 붙인다.

- **회의 전 — AI 아젠다 자동 생성**: 회의 시작 전 AI가 §1 섹션 ⑴~⑸ 구성의 아젠다 초안을 만들어 팀채널·회의 화면에 제공(마일스톤 안건은 간트 위험만 보기와 동기).
- **회의 중 — AI 라이브 진행(선택)**: §1-c/§3-b(b)의 순차 진행 인프라 재사용 — 아젠다 섹션을 하나씩 띄우고, 팀 라운드는 한 사람씩 호명. 대면이면 화면만 공유하고 진행은 사람이 해도 된다(아젠다 화면 자체가 가치).
- **회의 후 — 회의록 자동 초안**: 아젠다 + 회의 중 결정·입력을 묶어 회의록 초안 생성. **MeetingNote.kind 확장: `milestone` | `weekly` | `general`** — 주간 회의록만 모아보기. Action 추출→Task 파이프라인은 기존 그대로.
- **회의록 캡처 방식(2026-07-11 재확정 — Plaud 채택)**:
  ⑴ **행동 기반 회의록 = 정본** — 회의 중 시스템 행동(마일스톤 드래그·담당자 변경·Action 지정·결제 승인·도움 요청·LLM 채팅 입력)이 회의록의 기준. 결정은 전부 여기 남는다.
  ⑵ **Plaud 녹음·전사 → 대조 검증** — 녹음·전사·요약은 **Plaud(AI 녹음기)가 전담**. 회의 후 Plaud 전사 MD를 기존 회의록 업로드 파이프라인에 올리면, AI가 **행동 기반 회의록과 대조**해 ⓐ전사에는 있는데 시스템에 안 남은 결정(누락) ⓑ서로 어긋나는 내용(불일치)을 플래그로 제시 → 사람이 확인·반영. 병합 자동 덮어쓰기가 아니라 **검증 게이트**다(행동 기반이 항상 정본).
  ⑶ STT 내장은 하지 않는다(Plaud로 대체 — 기획에서 제외).
- **회의 LLM 채팅 입력(2026-07-11 확정 — 신규 입력의 기본 인터페이스)**: 시스템에 아직 없는 것(신규 마일스톤·Task·Action 담당 지정·결정 메모)을 회의 중 등록할 때는 **폼이 아니라 LLM 채팅**으로 입력한다 — 진행 화면(주간 회의·마일스톤 회의·간트 페이지)에 채팅창 상주.
  - 예: "에픽게임즈 8월 정산 리포트 마일스톤 8/7로" → AI가 해석 → **확인 카드**(프로젝트·제목·기한) → 확정 시 core mutation 실행(ChangeLog via=web).
  - 기존 자연어 입력 파이프라인(parseTaskInput→확인 카드)의 확장이며, 실행 계층은 **MCP 도구 19개와 같은 core mutation을 재사용** — 채팅이 곧 "회의용 액션 콘솔"이 된다. 스탠드업 대화형 체크인(§3-b)과 같은 LLM 채팅 인프라를 공용화한다.
  - 로드맵: Phase 2.5(AI 회의 진행 묶음)에 포함 — 채팅 인프라는 §3-b(a)와 공동 개발.
- 로드맵: **Phase 2.5에 포함**(아젠다 생성=프리뷰·요약과 같은 생성 경로, 진행=공유 인프라, kind 확장은 Phase 1의 MeetingNote.kind에 값만 추가).

## 2. 데이터 모델 (신규 엔티티 3 + 필드 2)

원칙: **파생 데이터(어제/오늘/막힘)는 저장하지 않고 기존 `getStandupData`(apps/web/src/lib/team-data.ts:230) 파생을 계속 쓴다. 저장하는 것은 "사람이 쓴 말"뿐.** 단 파생은 다음 날 재현이 안 되므로 제출 시점 Task id만 경량 동결.

### standup_entries — 비동기 체크인
- `id · date(KST YYYY-MM-DD) · userId · focus(필수, ~200자 — 오늘의 포커스 한마디) · note(선택) · blockerText(선택) · blockedTaskIds(선택 — issue/on_hold Task 참조) · snapshotTaskIds{yesterdayDone, yesterdayUnfinished, todayPlanned}(id만 동결) · aiDrafted · draftEdited · submittedAt/updatedAt`
- 유니크 (date, userId) 1건 — 재제출은 덮어쓰기. ChangeLog 생략(운영 리듬 기록 — revision_notes 선례), updatedAt만 추적.

### standup_team_summaries — AI 팀 요약 (AI 저장 관례의 의도적 예외)
- 날짜당 1건: `date · generatedAt · model · content(막힘 클러스터/흐름/추천 액션) · submittedUserIds · regeneratedBy`
- 예외 사유: Slack 게시·보드 표시 시점 분리, 하루 1회 pro 재생성 비용, 과거 회고 재현. 스키마 주석에 예외 명시.

### objectives — 분기 목표 (회사 레벨 단일 계층, 팀/회사 분리는 과설계)
- `id · title · description · period("2026-Q3") · ownerId · status(draft|active|done|cancelled) · order`

### key_results — 월 핵심결과 (개인 담당)
- `id · objectiveId · title · ownerId · month("2026-07") · metricType(manual|task_auto) · targetValue/currentValue/unit(manual용) · status · updatedAt/updatedBy`
- **진척 변경은 ChangeLog 기록**(목표 데이터 = 감사 대상).

### 기존 엔티티 필드 추가
- `Task.keyResultId`(optional, 단일) — 다대다 테이블은 8인 규모에 과설계.
- `MeetingNote.kind`(`milestone` | `weekly` | `general`, 기본 general) — §1 회의 종류(주간 전체 회의 §1-f 포함).

### KR 측정 방식 — 하이브리드(추천 확정)
- KR 단위 `metricType` 선택, **기본 manual**. 진척 계산기는 하나: manual=currentValue/targetValue, task_auto=연결 Task done/전체(기존 `overallProgress` 술어 재사용). 소비처는 단일 progress(0~100)만 본다.
- 근거: 순수 자동이면 매출·계약 등 비Task 지표 표현 불가, 순수 수동이면 Task 연결 가치 소멸.

### ③-b AI 진행 모드 — 대화형 체크인 (2026-07-11 추가)

폼 입력 대신 **AI가 스탠드업 진행자가 되어 1문 1답으로 묻고, 답하면 다음으로 넘어가는** 모드. 마일스톤 회의 AI 원격 진행(§1-c)과 같은 패턴의 스탠드업판 — 안건 큐·순차 질문·즉시 반영 인프라를 공유한다.

**(a) 개별 대화형 (비동기 기본)** — 모델: flash
- 10:00 DM 또는 /daily 진입 시 AI가 그 사람의 데이터를 알고 **맞춤 질문**을 순차로 던진다:
  ⑴ "어제 '배너 시안 검토'를 끝내셨네요. 미완 2건('카피 교정' 등)은 오늘 이어가시나요?" → 답
  ⑵ "오늘의 포커스 한마디는?" → 답(focus)
  ⑶ "막힌 건 없나요?" → 답(있으면 어떤 작업인지·도움 필요한 사람 — 문제/홀드 상세 계약과 연결)
- 대화가 끝나면 AI가 답을 standup_entry로 정리해 **확인 카드**로 보여주고, 사용자가 제출을 눌러야 저장(자동 저장 금지 관례). 폼이 더 빠른 사람은 기존 폼 입력과 자유 선택.
- Slack DM에서도 같은 흐름(버튼+짧은 텍스트 답) — /daily에 안 들어와도 체크인 완료 가능.

**(b) 라이브 진행 (모이는 날)** — §1-c 인프라 재사용
- 전원이 /daily(또는 회의실 화면)에 접속한 상태에서 AI가 **한 사람씩 호명**: "다음은 리원님 — 오늘 포커스는?" → 본인이 답(입력 또는 구두 후 기록) → 화면에 카드 갱신 → 다음 사람.
- 진행 상태 표시("4/8 완료, 지금: ○○님"), 전원 종료 시 그 자리에서 AI 팀 요약(②) 생성·표시.
- 발언 순서는 막힘 있는 사람 우선(병목 먼저 드러내기) → 나머지 랜덤.

로드맵 위치: (a)는 **Phase 2에 포함**(개인 초안의 자연 확장 — 같은 flash 경로), (b)는 **Phase 2.5와 함께**(§1-c 순차 진행 인프라 공유).
