# Griff OS Whitepaper v1.0

> 2026-07-11 · GRIFF
> 이 문서는 기능 명세가 아니다. **그리프가 회사를 어떻게 운영하는가**에 대한 철학과 프레임워크다.
> 실행 기획은 `que+/company-os-plan.md`, 현재 라이브 기능은 `que+/daily-standup-okr-plan.md`를 본다.

---

## 1. 선언

> Griff Q는 프로젝트를 관리하는 툴이 아니다.
> **회사가 어떻게 움직이는지를 정의하는 운영체계(Operating System)다.**

궁극의 목표는 하나다 — **회사의 모든 운영 원칙이 Griff Q 안에서 정의되고, 실행되고, 기록되는 것.** 문서에 적힌 원칙과 실제 일하는 방식이 다르면 원칙은 죽는다. 원칙이 도구 안에 살아 있을 때만 문화가 된다.

---

## 2. 운영 철학 — 여섯 가지 원칙

### 원칙 1. 감시하지 않는다, 드러낸다
Que의 창립 원칙. 도구는 사람을 지켜보기 위해서가 아니라 **업무 병목과 일정 충돌을 빨리 드러내기 위해** 존재한다. 모든 모듈은 이 시험을 통과해야 한다: *"이 기능은 사람을 평가하는가, 일이 막힌 곳을 보여주는가?"*

### 원칙 2. 결과보다 대응
실패는 두 종류다 — 우리가 통제할 수 있었던 **내부 실패**(일정 관리·QA·커뮤니케이션·승인 누락)와 통제 불가능한 **외부 변경**(클라이언트 방향·예산·일정 변경, 행사 취소). 이 둘을 반드시 분리한다.

외부 변경은 실패가 아니다. **대응하지 못한 것이 실패다.** 그래서 우리는 결과가 아니라 대응을 관리한다: 변경 발생 → 24시간 내 영향 분석 → 재협의 → 승인. 이 프로세스를 탄 실패는 "관리된 실패"이고, 관리된 실패는 부끄러운 것이 아니라 **자산**이다.

### 원칙 3. Task ≠ Key Result
Task는 행위다 — 디자인한다, 개발한다, 퍼블리싱한다. Key Result는 **결과가 어떻게 되었는가**다. 에이전시의 성과는 MAU가 아니라 납품이므로, 우리의 KR은 수치가 아니라 **상태(State)** 로 정의한다: 일정 내 납품, 클라이언트 승인, 오류 없이 전달. 열심히 일한 것과 결과가 좋은 것을 구분하는 힘이 여기서 나온다.

### 원칙 4. 직무는 유지되고, 책임은 환경을 따른다 (Role & Arena)
**Role(직무)** 은 쉽게 변하지 않는다 — PM, Designer, Developer, AE, Producer. 그러나 **Arena(업무 환경)** 가 바뀌면 같은 사람의 책임이 달라진다. 사무실의 디자이너는 시안을 만들지만, 행사장의 디자이너는 출력물을 관리하고 현장에서 긴급 수정을 한다.

Arena는 사람을 평가하기 위한 개념이 아니다. **적재적소에 배치하기 위한 기준**이다. 우리는 사람에게 점수를 매기지 않는다 — 강점을 기록하고, 강점이 필요한 곳에 배치한다.

### 원칙 5. 프로젝트는 Task만으로 움직이지 않는다
운영에는 Task 외에 반드시 존재해야 하는 것들이 있다 — **Decision Rule**(누가 무엇을 결정하는가), **Change Management**(변경은 어떻게 접수·전파되는가), **Quality Gate**(무엇을 통과해야 나가는가), **Risk**(무엇이 일정을 위협하는가), **Finance**(이 일은 남는 장사인가). 이들이 도구 밖(머릿속·채팅방)에 있으면 회사는 사람 기억력으로 돌아간다.

### 원칙 6. 기록되지 않은 결정은 없었던 결정이다
모든 업무 영향 변경은 누가·언제·무엇을 바꿨는지 남는다(ChangeLog). 회의는 산출물(회의록·Task·결정)로 끝난다. AI가 보고를 대신 만들고, 사람은 결정만 한다 — 그리고 그 결정이 기록이다.

## 3. 프레임워크 — Griff OS 구조

```
Griff OS
│
├── Griff Q (Project Core)          ← 지금 라이브. 건드리지 않고 유지한다
│     Project · Workflow · Task · Schedule · Approval
│
├── HR OS        Role · Arena · Skill(강점 태그) · 배치
├── Finance OS   Budget · Cost · Profit · Resource
├── Knowledge OS Wiki · Template · SOP · Lessons Learned
├── Governance OS Decision · Approval · Policy · Authority(R&R)
└── Product OS   Vision · Roadmap · Release · Feedback · Growth
```

**확장 원칙 — Griff Q Plus**: 기존 Griff Q(Project Core)는 그대로 두고, 모듈을 옆에 붙인다. 코어를 다시 만들지 않는다. 각 모듈은 코어의 데이터(작업·프로젝트·기록)를 소비하며 자란다 — HR의 배치 이력은 작업 데이터에서, Finance의 비용은 결제요청에서, Knowledge의 사례는 회고에서 파생된다.

**Agency와 Product의 분리**: 에이전시는 남의 문제를 해결하고(성과=납품), 제품은 우리의 문제를 해결한다(성과=성장). 두 운영 방식을 같은 화면·같은 지표로 섞지 않는다. Product OS는 자체 제품이 운영 단계에 들어갈 때 별도 트랙으로 세운다.

## 4. 지금 어디에 있는가

| 층 | 상태 |
|---|---|
| Project Core (PM) | ✅ 라이브 — 프로젝트·작업·마일스톤·간트·일정 |
| Operation 리듬 | ✅ 라이브 — 데일리 스탠드업·주간 통합 회의·긴급 결정·AI 진행 |
| 목표(OKR) | ✅ 라이브(수치·작업연동) — **상태형 KR이 다음** |
| Governance | ◐ 기록은 있다(ChangeLog·승인) — 실패 분류·대응 프로세스가 다음 |
| HR (Role & Arena) | 설계 완료, 미구현 |
| Finance | 결제요청만 — 프로젝트 손익이 다음 |
| Knowledge | 회고 데이터가 쌓이면 승격 |
| Product OS | 미착수(자체 제품 성숙 시) |

로드맵 상세는 `company-os-plan.md`의 OS-1~OS-6.

## 5. 이 문서를 읽는 법

- 새 기능을 제안할 때: **§2의 여섯 원칙에 비춰라.** 원칙과 충돌하면 기능이 아니라 원칙을 논의 안건으로 올린다.
- 새 팀원이 왔을 때: 이 문서가 "우리 회사는 이렇게 일한다"의 첫 페이지다.
- 분기마다: §4의 현재 위치를 갱신한다. 화이트페이퍼는 완성본이 아니라 **버전이 올라가는 문서**다.
