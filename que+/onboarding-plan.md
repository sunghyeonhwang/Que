# Que 베타 온보딩 계획 — "복잡하지 않게 들어오기" (2026-07-14)

> 배경: 사용자(대표) 질문 — "사원들이 쓰기에 이 툴이 너무 복잡하지 않은가? 처음부터 너무 많은 걸 요구하나?"
> 결론 먼저: **도구는 그대로 두고, 요구를 주차별로 나눈다.** 첫 주에 팀원에게 요구하는 것은 딱 3가지다.

---

## 1. 진단 — 복잡한가?

정직하게: **메뉴 기준으로는 복잡하다**(메뉴 15개+, 개념 10여 개). 하지만 팀원의 하루 기준으로 보면 다르다:

- 팀원이 **매일** 쓰는 것: 데일리 체크인(30초) · 작업 완료 체크 · (필요 시) Copilot 질문 — **3가지뿐**
- **매주** 쓰는 것: 월요일 회의 화면(보기만) · 내 작업 등록
- 나머지(성과·회의록·반복·결제·클라이언트)는 **필요할 때만** 열거나 **관리자 전용**

즉 문제는 기능의 수가 아니라 **"어디부터인지 모른다"는 첫인상**이다. 온보딩은 이 첫인상을 관리하는 일이고, 도구를 덜어낼 필요는 없다(안 쓰는 메뉴는 안 열면 그만이고, 필요해질 때 그 자리에 있어야 한다).

## 2. 원칙

1. **첫 주에는 습관 하나만**: 도구를 배우는 게 아니라 "10시 체크인" 습관 하나를 붙인다. 도구는 습관 뒤에 따라온다.
2. **가르치지 말고 오게 한다**: 교육 세션 대신 Slack 알림이 데일리로 데려온다(버튼 → 딥링크). 배울 필요가 없는 진입.
3. **역할별로 요구가 다르다**: 팀원 3가지 / 관리자 +2가지 / 대표 +1가지. 전원에게 같은 걸 요구하지 않는다.
4. **질문은 Copilot에게**: "이거 어디서 해?"를 사람에게 묻지 않게 — Copilot과 도움말·설계 FAQ가 1차 창구.

## 3. 주차별 로드맵

### 0주차 — 발표일(베타 전, 이번 주)
- 전체 발표(PPT) 1회: "왜 만들었나 + 하루가 어떻게 달라지나"만. 기능 나열 금지.
- 쇼츠 #01(체크인) 1편만 미리 공유. QA 시트로 전원 1회 둘러보기(각자 로그인·비번 확인 겸).

### 1주차 (7/20~) — 요구 3가지만
| 대상 | 요구 | 요구하지 않는 것 |
|---|---|---|
| 전원 | ① 10시 체크인(30초) ② 내 작업 끝나면 완료 체크 ③ 막히면 막힘 칸에 쓰기 | 작업 등록 정리, OKR, 회의록, 성과 확인 |
| 관리자 | + 월요일 회의 진행 모드로 주간 회의 1회 | 리포트 분석 |
| 대표 | + 홈에서 아침 조망(보기만) | — |

- **Slack에 쇼츠 하루 1편**("오늘의 Que 한 가지", #01→#13 순서) — 각 30초, 강요 없음.
- 금요일: 수정사항(/revisions)에 쌓인 불편 리뷰 → 주말 핫픽스.

### 2주차 — 각자 일 붓기
- 자기 작업을 자연어로 등록(작업 목록 입력 탭 또는 Copilot)하기 시작.
- OKR 탭 오픈: 분기 목표·KR을 회의에서 함께 등록(개인 숙제 아님).
- 마일스톤·반복 업무는 담당자(관리자·PM)만.

### 3주차~ — 자연 확장
- 성과·히트맵을 주간 회의에서 함께 보기(개인 평가 아님을 그 자리에서 재확인).
- 결제요청·회의록 업로드 등은 해당 업무가 생길 때 그 사람에게만.

## 4. 제품에서 받쳐줄 것 (구현 제안 — 지시 시 진행)

1. **홈 온보딩 카드(1~2주만 노출)**: 첫 로그인 시 홈 상단에 "이번 주는 이것만" 카드 — ①10시 체크인 ②완료 체크 ③막힘 쓰기, 각 항목에 해당 화면 딥링크+완료 여부 체크. 2주 후 자동 소멸(또는 닫기). *난이도: 소~중.*
2. **빈 화면을 교육으로**: 이미 상당수 화면에 교육형 빈 상태가 있음(간트 프라이머·OKR 설명 등) — 유지·보강.
3. **설계 FAQ(/faq)**: "왜 이렇게 만들었나" 질문 창구 — 구현 완료(이 배치로 배포). 새 질문이 나올 때마다 여기 축적.
4. **쇼츠 13편**: Slack 배포용 준비 완료(`Powerpoint/videos/shorts/`).

## 5. "너무 많이 요구하나?"에 대한 직답

첫 주 기준으로 팀원 1인에게 요구하는 시간은 **하루 1분 미만**(체크인 30초 + 완료 체크 몇 번)이다. 이것이 대체하는 것은 아침 보고·"그거 어떻게 됐어?" 메시지·주간 보고 정리다. 요구가 아니라 교환이고, 교환비가 남는다는 걸 첫 주 안에 체감하게 만드는 것이 온보딩의 전부다. 체감에 실패하면(1주차 금요일 수정사항·체크인 제출률로 판정) 2주차 확장을 멈추고 1주차를 반복한다.
