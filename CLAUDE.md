# Que

Que는 캘린더 UI를 가진 팀 작업 상태 관리 도구다. 8명 회사 팀이 회사 캘린더, 개인 작업, 프로젝트 마일스톤, 회의록 Action, 결제 요청, 업무량 현황을 확인하고 수정한다. 감시 도구가 아니라 업무 병목과 일정 충돌을 빨리 드러내는 운영 도구로 구현한다.

## Source of truth

- 기획: `data/docs/que-product-plan.md` (유일한 기획 기준. 요약본은 없음 — 다시 만들지 말 것)
- 디자인: `data/DESIGN.md` (shadcn/ui 기본값 기준)
- MCP/CLI 계획: `data/docs/que-mcp-cli-plan.md`
- 개발 프롬프트: `data/docs/claude-prompts/`
- 디자인/정보 구조 레퍼런스: `data/preview/*.html` (React 컴포넌트는 shadcn/ui 기준으로 새로 구성)
- 세션 간 인수인계: `HANDOFF.md` — 의미 있는 결정이 생기면 반드시 갱신한다

## 스택

TypeScript · Next.js App Router · Tailwind CSS · shadcn/ui (Radix 접근성 유지) · lucide-react · react-hook-form + zod · mock data 우선(API/DB로 교체 가능한 구조)

## 메뉴 구조 (확정, 2026-07-03 재편)

라벨은 아래로 바꾸되 **URL은 유지**한다(링크·쿠키·테스트 영향 최소화). 병합은 URL 기반 탭 전환.

- **작업 목록** (`/today`) — 오늘·Now 탭 병합. 오늘=팀원 기본 진입점(개인), Now=관리자·프로젝트 담당자 기본 진입점(팀 운영표). 두 화면 상단에 한 줄 역할 설명 고정. 루트(`/`) 역할별 리다이렉트 유지.
- **프로젝트** (`/calendar`) — 캘린더 **단일 메뉴**. 기본형/전체 멤버/타임라인 뷰 스위처(타임라인은 가로/세로 + 시간별/1주간/2주간). 별도 메뉴 금지, 마지막 뷰 기억, URL 반영.
- **마일스톤** (`/projects`) — 마일스톤·연결 작업·반복 업무 템플릿.
- **일정** (`/team`) — 팀 현황(운영 보드/스탠드업/[관리자]리포트 뷰 토글).
- **퍼포먼스** (`/heatmap`) — 히트맵. 멤버별 작업량 편차(평가 아님).
- **팀** (`/members`) — 팀원 카드 + 업무 요약(조회 전용). 프로필 편집/PII(전화·생년월일)는 미포함.
- **회의록** (`/meeting-notes`) — 회의록·Action 탭 병합.
- **결제요청** (`/payments`) — 결제 요청·입금 상태(계좌·금액 마스킹).
- **홈** — 전체현황 대시보드. 사용자가 디자인 별도 제공 예정 → **미착수**.
- 알림 · 설정 — **개발 후순위** (프리뷰 별도 제공 전 착수 금지)

## 화면 원칙

- 태블릿 우선. 검수 해상도: FHD 1920x1080, 1366x768, 태블릿 가로 1024x768, 태블릿 세로 768x1024
- 태블릿 세로에서 사이드바는 접히거나 상단/하단 내비게이션으로 전환
- 캘린더/히트맵/테이블은 내부 스크롤 + sticky header/label. 페이지 전체를 깨지 않는다
- 터치 대상(버튼/행/드래그)은 최소 40px, 가능하면 44px 이상
- 운영 도구 톤: 히어로·장식 그래픽·카드 중첩 금지. 표, 상태 배지, 액션 버튼이 먼저 보이게
- 상태 색상 의미 고정: green=진행/완료, blue=예정/정보, amber=주의/대기, red=문제/취소, violet=회의록/응답대기
- shadcn 글로벌 theme token을 브랜드 색으로 덮어쓰지 않는다

## 도메인 규칙 (서버/로직에서 강제)

- 담당자 또는 마감일 없는 Action은 Task로 자동 생성하지 않는다 (`확인 필요`로 남김)
- 외부 회사 캘린더 원본 일정과 비공개 자리비움은 드래그/수정 불가
- `문제발생`/`홀드`는 사유·다음 액션·도움 필요한 사람·재확인 시간을 함께 받는다
- 자연어 입력은 등록 전 확인 카드를 반드시 거친다
- 업무에 영향을 주는 변경은 ChangeLog에 기록 (누가/언제/무엇을, `via: web|mcp|cli`)
- 본인 작업만 수정 가능. 타인 작업은 댓글/도움 요청/상태 확인 요청만

## 작업 방식

1. 레포 구조를 읽고 계획을 제안한 뒤, 승인받으면 단계별 구현
2. App Shell → mock data → 핵심 페이지 → 상호작용 → 검증 순서
3. 구현 후 lint, typecheck, build, 브라우저 검증 결과를 보고
4. 데이터 접근은 처음부터 core 계층으로 분리 (MCP/CLI가 같은 계층을 재사용할 예정)
5. **dev 서버(`pnpm dev`)가 떠 있는 동안 `pnpm build`를 실행하지 않는다.** 둘이 같은 `.next`를 공유해 dev 서버 캐시가 깨진다. build 전에 dev 서버를 종료하고, 검증 후 다시 띄운다.
