# Que

Que는 캘린더 UI를 가진 팀 작업 상태 관리 도구다. 8명 회사 팀이 회사 캘린더, 개인 작업, 프로젝트 마일스톤, 회의록 Action, 결제 요청, 업무량 현황을 확인하고 수정한다. 감시 도구가 아니라 업무 병목과 일정 충돌을 빨리 드러내는 운영 도구로 구현한다.

## Source of truth

- 기획: `data/docs/que-product-plan.md` (유일한 기획 기준. 요약본은 없음 — 다시 만들지 말 것)
- 디자인: `data/DESIGN.md` (shadcn/ui 기본값 기준)
- MCP/CLI 계획: `data/docs/que-mcp-cli-plan.md`
- 개발 프롬프트: `data/docs/claude-prompts/`
- 디자인/정보 구조 레퍼런스: `data/preview/*.html` (React 컴포넌트는 shadcn/ui 기준으로 새로 구성)
- 세션 간 인수인계: `HANDOFF.md` — 의미 있는 결정이 생기면 반드시 갱신한다

## 스택

TypeScript · Next.js App Router · Tailwind CSS · shadcn/ui (Radix 접근성 유지) · lucide-react · react-hook-form + zod · mock data 우선(API/DB로 교체 가능한 구조)

## 메뉴 구조 (현행, 2026-07-04 재설계 배포 IA — `apps/web/src/lib/menu.ts` 정본)

전면 재설계로 IA가 아래로 확정됐다. **이 절이 `menu.ts`와 항상 일치해야 한다**(불일치가 고아 라우트를 낳음). 작업목록·회의록은 URL 기반 탭 병합.

- **홈** (`/home`) — 개인 대시보드(KPI·히트맵·오늘 할 일·개인 일정·작업 분포). 성과 컴포넌트 재사용.
- **일정** (`/schedule`) — 주/월 캘린더 + 날짜 이동. (구 `/calendar`는 은퇴 → `/schedule`로 redirect. 팀 캘린더 드래그 재일정은 팀 현황/오늘의 상태 시트로 유지.)
- **성과** (`/heatmap`) — KPI·완료율·기한초과·히트맵(월 그리드)·저성과표·프로젝트 진행률. 기간/월 선택 URL 구동. 개인 평가 아님.
- **작업 목록** (`/today`, `/now` 탭 병합) — 오늘=팀원 개인 진입, Now=관리자·PM 팀 운영표. 루트(`/`) 역할별 리다이렉트 유지.
- **팀** (`/members`) — 팀원 카드 + 부서/요약(**조회 전용**). 멤버 추가/편집/제거·PII(전화·생년월일)는 미포함(미구현 데모 컨트롤은 미노출).
- **팀 현황** (`/team`) — 운영 보드/스탠드업/[관리자]리포트 뷰 토글. 관리자·팀장 상시 사용 핵심 운영 화면.
- **회의록** (`/meeting-notes`, `/action` 탭 병합) — 회의록·Action. 상단 요약 카드, 사이드바 뱃지=확인 필요 수. (2026-07-05 메뉴 라벨 '확인필요'→'회의록', 아이콘 FileText. `/meeting-notes` 화면 제목은 '회의록', `/action` 탭 화면 제목은 성격상 '확인필요' 유지.)
- **반복·마일스톤** (`/planning`) — 정기 반복 업무 템플릿(등록/켜끄기, 회차 Task 자동 생성) + 프로젝트 마일스톤(생성/수정/위험표시, 담당자·관리자만). 기존 완성 백엔드(core mutation·스케줄러·영속) 재연결. 태스크-마일스톤은 프로젝트 공유로 간접.
- **프로젝트** (`/projects`) — PM 도구. 보드(칸반)/목록/캘린더 뷰 + 카드 상세 드로어. **카드 = core Task**(담당자 단수), **컬럼 = status 고정 4열**(예정/진행중/홀드·문제/완료 — cancelled·merged 제외). 열 이동=`changeTaskStatus`(홀드·문제 열은 사유 필수). 좌측 프로젝트 목록은 활성 클라이언트 필터 스코프 존중. **전원 노출**(adminOnly 아님) — 쓰기는 core `canEditTask`가 카드 단위로 강제. 생성/편집/삭제(취소 soft)/이동 모두 core mutation 경유 + ChangeLog via=web. DB화 완료로 프리뷰 배너·`QUE_PM_WRITE` 게이트 제거(2026-07-05).
- **클라이언트** (`/clients`) — **관리자 전용**. 거래처(클라이언트) + 소속 프로젝트 관리(클라이언트 추가/이름수정/보관, 프로젝트 추가/편집·클라이언트 재배정·상태·담당자). menu.ts `adminOnly` 플래그로 비관리자에게 숨기고, 페이지·서버 액션·core mutation(canManageClient/canManageProject)이 3중으로 강제. core mutation 경유 + ChangeLog via 기록.
- **결제요청** (`/payments`) — 결제 요청·입금 상태(계좌·금액 마스킹, 권한별 노출).
- **수정사항** (`/revisions`) — 기타 섹션. 테스트 중 발견한 수정사항(이슈/피드백) 팀 공용 목록. 필드: 메뉴·위치·오류사항·상태(미해결/보류/해결)·작성자·시간. **전원 접근(관리자 전용 아님), 누구나 작성·상태 변경 가능**(인증만). 업무 데이터가 아니라 ChangeLog 없이 updatedAt/updatedBy만 추적. core mutation(createRevisionNote/updateRevisionNoteStatus) 경유.
- **MCP · CLI** (`/tools`) — 기타 섹션. 터미널(CLI)·AI(MCP) 연결 온보딩(내 계정·PAT 안내·복사 가능한 설정 블록·Claude Code/Desktop/Gemini CLI 등록법). 조회 전용, 실 PAT 미노출. 전체 레퍼런스는 `data/docs/que-tools-guide.md`.
- **도움말** (`/help`) — 기타 섹션. 비개발 팀원용 사용 설명서(화면별 사용법·자주 하는 일 따라하기·규칙/색상 의미·FAQ·문제해결). 왼쪽 섹션 목차 + 검색. **매뉴얼 톤(설명 '합니다'·지시 '하세요', 구어체 금지)** — 2026-07-05 '쉬운 해요체'에서 변경. 섹션은 menu.ts 메뉴와 매칭(회의록·결제요청·반복마일스톤·클라이언트 각각 독립 섹션).

## 화면 원칙

- 태블릿 우선. 검수 해상도: FHD 1920x1080, 1366x768, 태블릿 가로 1024x768, 태블릿 세로 768x1024
- 태블릿 세로에서 사이드바는 접히거나 상단/하단 내비게이션으로 전환
- 캘린더/히트맵/테이블은 내부 스크롤 + sticky header/label. 페이지 전체를 깨지 않는다
- 터치 대상(버튼/행/드래그)은 최소 40px, 가능하면 44px 이상
- 운영 도구 톤: 히어로·장식 그래픽·카드 중첩 금지. 표, 상태 배지, 액션 버튼이 먼저 보이게
- 상태 색상 의미 고정: green=진행/완료, blue=예정/정보, amber=주의/대기, red=문제/취소, violet=회의록/응답대기
- shadcn 글로벌 theme token을 브랜드 색으로 덮어쓰지 않는다

## 도메인 규칙 (서버/로직에서 강제)

- 담당자 또는 마감일 없는 Action은 Task로 자동 생성하지 않는다 (`확인 필요`로 남김)
- 외부 회사 캘린더 원본 일정과 비공개 자리비움은 드래그/수정 불가
- `문제발생`/`홀드`는 사유·다음 액션·도움 필요한 사람·재확인 시간을 함께 받는다
- 자연어 입력은 등록 전 확인 카드를 반드시 거친다
- 업무에 영향을 주는 변경은 ChangeLog에 기록 (누가/언제/무엇을, `via: web|mcp|cli`)
- 본인 작업만 수정 가능. 타인 작업은 댓글/도움 요청/상태 확인 요청만

## 작업 방식

1. 레포 구조를 읽고 계획을 제안한 뒤, 승인받으면 단계별 구현
2. App Shell → mock data → 핵심 페이지 → 상호작용 → 검증 순서
3. 구현 후 lint, typecheck, build, 브라우저 검증 결과를 보고
4. 데이터 접근은 처음부터 core 계층으로 분리 (MCP/CLI가 같은 계층을 재사용할 예정)
5. **dev 서버(`pnpm dev`)가 떠 있는 동안 `pnpm build`를 실행하지 않는다.** 둘이 같은 `.next`를 공유해 dev 서버 캐시가 깨진다. build 전에 dev 서버를 종료하고, 검증 후 다시 띄운다.
